In November 2016, Google switched Google Translate to a deep neural machine‑translation system and, overnight, translation quality jumped more than it had in the previous ten years combined. That single leap helped reignite the world's interest in machine learning (ML). In just a few years, ML has quietly moved into how we search, type, shop, unlock our phones, and even how some of us find love.

But here's the trap most newcomers fall into. When people hear *"machine learning system,"* they picture an algorithm — logistic regression, a neural network, maybe a transformer. **The algorithm is only a sliver of a real ML system.** A production system also includes the business reason the project exists, the interface where people use it, the data stack that feeds it, and the logic and infrastructure for developing, deploying, monitoring, and updating models.

## When to Use Machine Learning

ML is powerful, but it is not magic, and it is not always the *right* or *cost‑effective* tool. Before starting a project, it pays to ask a blunt question: **do we actually need ML here?** A useful working definition makes the answer clearer:

> Machine learning is an approach to **(1) learn (2) complex patterns** from **(3) existing data** and use those patterns to make **(4) predictions** on **(5) unseen data.**

Each italicized phrase is a condition. If your problem doesn't satisfy it, ML may be the wrong tool. The diagram below contrasts how traditional software and ML arrive at answers — the reason Andrej Karpathy nicknamed ML *"Software 2.0."*

### The five core conditions

**1. Learn — the system must be able to learn.** A relational database isn't ML: you can *state* the relationship between two columns, but it can't *figure one out* on its own. For ML to learn, there must be something to learn from — usually data. In supervised learning, the system sees example input–output pairs (an Airbnb listing → its nightly price) and learns to produce outputs for new inputs.

**2. Complex patterns — there must be patterns, and they must be complex.** ML only helps when there's a pattern worth learning.

::: callout analogy "Lookup table vs. learned recipe"
Sorting house listings into states by zip code needs **no ML** — each zip maps to exactly one state, so a *lookup table* is perfect. But predicting a listing's *price* from its size, location, amenities, and reviews follows a tangled pattern no human can write down by hand. That's the sweet spot for ML: instead of telling the system *how* to compute the price, you show it many prices and let it infer the recipe.
:::

There may be a pattern you can't see (does Elon Musk's tweeting move crypto prices?) — and you won't know until you've rigorously trained and tested a model. Failing to find a pattern doesn't prove there isn't one. Note too that *complex to a machine* differs from *complex to a human*: raising a number to the 10th power is trivial for a computer but tedious for us, while spotting a cat in a photo is the reverse.

**3. Existing data — data is available, or you can collect it.** ML learns from data, so there must be data. Predicting someone's tax bill is a fine idea — but impossible without income and tax records. A few nuances:
- **Zero‑shot learning** lets a model handle a task it wasn't trained on — but only because it was trained on *related* tasks first. It still needed data.
- With **continual learning**, a model can be deployed cold and learn from live traffic — at the risk of a poor early experience.
- Many teams use a **"fake‑it‑till‑you‑make‑it"** launch: serve human‑made predictions at first, and use the data collected to train a model later.

**4. Predictions — it must be a predictive problem.** ML produces predictions (estimates), so it only fits problems whose answers are predictive. It shines when you can benefit from *many cheap, approximate* answers. Increasingly, hard problems get *reframed* as prediction: rather than running an expensive simulation, ask *"what would the result of this simulation look like?"* and approximate it — a trick used heavily in graphics (image denoising, screen‑space shading).

**5. Unseen data shares the pattern.** The patterns learned from past data only help if *future* data looks similar — in technical terms, training and unseen data should come from **similar distributions.**

::: callout analogy "Studying last year's exam"
Training a model is like studying past exams. It works only if this year's exam covers similar material. A model trained on 2008 app‑store behavior (when *Koi Pond* was a top app) will flop predicting 2020 downloads — the world moved on. We can't *know* the future distribution, so we *assume* tomorrow resembles today, and watch closely (Chapters 8–9) for when that assumption breaks.
:::

### The four conditions where ML especially shines

Because of how today's algorithms learn, ML is *especially* worthwhile when the problem is also:

- **6. Repetitive.** Most algorithms need many examples to learn a pattern. Repetition means each pattern recurs often, making it easier to learn. (Humans few‑shot learn from a couple of cat photos; machines usually need thousands.)
- **7. Cheap when wrong.** Your model *will* make mistakes. ML fits best when a wrong prediction is forgiving — a bad Netflix recommendation just goes unclicked. High‑stakes uses (self‑driving cars) can still qualify *if*, on average, the benefit of correct predictions outweighs the cost of wrong ones.
- **8. At scale.** ML needs up‑front investment, so it pays off when you make *a lot* of predictions (sorting millions of emails, routing thousands of tickets). A "single" prediction is often many in disguise — an election forecast is re‑run constantly as new data arrives. Scale also means more data to learn from.
- **9. Constantly changing.** Tastes, cultures, and spam tactics shift. Hand‑written rules go stale fast and are painful to update. Because ML learns from data, you can refresh it with new data without manually re‑deriving the rules — and you can even let it adapt automatically (continual learning).

### When *not* to use ML

Even when ML *can* work, skip it (or postpone it) if any of these hold:

::: callout warning "Don't reach for ML when…"
- **It would be unethical.** (Chapter 11 covers a case study of an automated grader's biases.)
- **A simpler solution does the trick.** Chapter 6's model‑development phases start with *non‑ML* baselines for a reason.
- **It isn't cost‑effective.**
:::

Two caveats. First, even if ML can't solve the *whole* problem, it can often solve a *piece* of it — you may not be able to build a chatbot that answers every query, but you can build a model that predicts whether a query matches an FAQ. Second, don't dismiss ML just because it isn't cost‑effective *today*. Most technology advances incrementally; wait for it to prove itself to everyone else and you may end up years behind your competitors.

### Machine Learning Use Cases

ML has exploded across both **consumer** and **enterprise** applications since the mid‑2010s. On the consumer side you meet it constantly: recommender systems (Amazon, Netflix) and search engines that help you find things; predictive typing; photo‑enhancement suggestions; face/fingerprint unlock; machine translation that erases language barriers; smart assistants (Alexa, Google Assistant); security cameras that recognize your pets; and at‑home health monitors that detect falls.

Yet **most real ML value is in the enterprise**, where the requirements differ sharply from consumer apps:

| Dimension | Consumer apps | Enterprise apps |
|---|---|---|
| **Accuracy** | More forgiving (95% → 95.5% is unnoticeable) | Often strict (a 0.1% efficiency gain can save millions) |
| **Latency** | Critical — a 1‑second delay loses attention | Often more tolerant of higher latency |
| **Business model** | Easy to distribute, hard to monetize | Harder to distribute, easier to monetize |

Common enterprise use cases include **fraud detection** (anomaly detection on transactions — one of the oldest applications), **price optimization** (dynamic pricing for ads, flights, ride‑sharing), **demand forecasting** (stock the right inventory), **customer‑acquisition cost reduction** and **churn prediction** (keeping a customer is 5–25× cheaper than winning a new one), **support‑ticket classification** (route tickets to the right team automatically), **brand monitoring** via sentiment analysis, and a fast‑growing wave of **health‑care** applications (detecting skin cancer, diagnosing diabetes) — usually delivered through providers because of strict accuracy and privacy needs.

## Understanding Machine Learning Systems

To design ML systems well, you have to understand how they differ from the two things newcomers usually compare them to: **ML in research** (how it's taught in school) and **traditional software.**

### Machine Learning in Research Versus in Production

Most people with ML expertise learned it in academia, where the goals are very different from production. There are five major differences:

| | Research | Production |
|---|---|---|
| **Requirements** | State‑of‑the‑art on benchmark datasets | Different stakeholders, different (often conflicting) requirements |
| **Computational priority** | Fast training, high throughput | Fast inference, low latency |
| **Data** | Clean, static, well‑understood | Messy, shifting, possibly biased |
| **Fairness** | Often an afterthought | Must be considered |
| **Interpretability** | Often not a focus | Often a requirement |

#### Different stakeholders and requirements

In research, everyone usually aligns on **one** metric: beat the benchmark. In production, many stakeholders each pull in different directions. Consider an app that recommends restaurants and earns a 10% fee per order:

- **ML engineers** want a more complex model with more data to maximize relevant recommendations.
- **The sales team** wants it to favor *expensive* restaurants (bigger fees).
- **The product team** has measured that every bit of added latency drops orders, so they demand results in **under 100 ms.**
- **The ML platform team**, woken up at night by scaling issues, wants to *freeze* model updates to stabilize the platform.
- **The manager**, eyeing margins, might even consider letting the ML team go.

"Recommend what users will click" and "recommend what makes the most money" are *two different objectives*. (Chapter 2's **decoupling objectives** shows the fix: train one model per objective and combine them.) The key skill is understanding every stakeholder's requirements and how strict each one is — a 100 ms cap is very different as a *must‑have* than as a *nice‑to‑have*.

::: callout note "Why great research models often don't ship"
**Ensembling** — combining many models — wins Kaggle competitions and famously took the \$1 million Netflix Prize. Yet it's rarely used in production: it makes systems too complex, slower, and harder to maintain for a small accuracy gain. Critics also note that benchmark leaderboards reward accuracy at the expense of qualities practitioners value — compactness, fairness, and energy efficiency — and that with many teams testing on the same hold‑out set, a winner can come out ahead **by chance** (multiple‑hypothesis testing).
:::

#### Computational priorities

A common rookie mistake is to obsess over model *development* and neglect *deployment and maintenance*. During development, you train many models and each does many passes over the data — **training is the bottleneck.** Once deployed, the model's job is to serve predictions — **inference is the bottleneck.** So research optimizes **fast training / high throughput**, while production optimizes **fast inference / low latency.**

- **Latency** = time from receiving a query to returning its result.
- **Throughput** = how many queries you handle per unit time.

::: callout note "A terminology note on latency"
Some texts separate *latency* (time a request waits to be handled) from *response time* (what the client actually experiences, including network and queueing). To match common ML usage, this book uses **latency** to mean the full response time — from when a request is sent to when the response arrives.
:::

Processing one query at a time, higher latency means lower throughput (10 ms/query → 100 queries/sec). But modern systems **batch** queries, so higher latency can *also* mean higher throughput: process 50 queries in a 20 ms batch and you get 2,500 queries/sec even though each query waits 20 ms. The catch with batching *online* traffic is that you must wait for enough queries to fill a batch, which adds latency.

Crucially, **latency is a distribution, not a single number** — and the average lies.

::: callout analogy "Bill Gates walks into a bar"
Average ten regular incomes and one billionaire's, and the "average income" suggests everyone's rich. One outlier drags the mean. Latency is the same: ten requests at ~100 ms and one stuck at 3,000 ms average out to 390 ms — making your *fast* system look sluggish. That's why engineers think in **percentiles**, not means.
:::

The **p50 (median)** is the middle value; half your requests are faster, half slower. Higher percentiles — **p90, p95, p99** — expose the painful tail. Those slow requests matter more than their count suggests: on Amazon, the customers with the *slowest* requests are often those with the *most data* — i.e., the most valuable customers. Teams routinely write requirements like *"p99 latency must stay below X."*

#### Data

In research, datasets are usually **clean, well‑formatted, and static** — many people have used them, the quirks are known, and you can grab a script to feed them straight into a model. In production, data is **messy, noisy, possibly unstructured, and constantly shifting.** It's likely biased (and you may not know how), labels may be sparse or wrong, and shifting business needs may force you to relabel. If it's user data, privacy and regulation enter the picture. You also move from working with *historical* data to data being **generated live** by users and systems.

#### Fairness

In research it's tempting to "get state of the art first and worry about fairness later." By production, it's too late — and there's no agreed "state of the art" for fairness metrics to chase. ML doesn't predict the future so much as **encode the past**, perpetuating its biases — and at scale it can discriminate against millions in a split second. Real harms documented in the book include creditworthy Black and Latino applicants being rejected for loans (acceptance flipped when race identifiers were removed), biased résumé ranking, and credit‑score effects that punish the poor. As of the book's writing, only a small minority of large companies were actively mitigating algorithmic bias — though that's changing. (Chapter 11 covers responsible AI in depth.)

#### Interpretability

Geoffrey Hinton's provocation: would you accept a *black‑box* AI surgeon with a **90%** cure rate over a human with **80%** — knowing it can't explain itself? When the author posed this to 30 tech executives, only *half* would take the better‑but‑inscrutable AI. Interpretability isn't optional in industry — it's often a requirement: users and leaders need to **trust** decisions (and spot bias), and developers need it to **debug and improve** models. Yet because research still optimizes a single metric (performance), researchers aren't incentivized to work on it.

#### Discussion: where the jobs are

It's fine to know only the academic side of ML — but the claim that "there are plenty of research jobs, so that's enough" is false. Pure research is expensive; most companies can only fund it when it leads to near‑term applications. As off‑the‑shelf models get more accessible, demand shifts toward **applying** them. The vast majority of ML jobs are — and will be — in *productionizing* ML.

### Machine Learning Systems Versus Traditional Software

Since ML is part of software engineering, why not just reuse 50 years of SWE best practices? You should — many tools transfer, and ML production would be far healthier if ML practitioners were better software engineers. But ML breaks one core SWE assumption: that **code and data are separate.** SWE prizes separation of concerns; an ML system is **part code, part data, and part artifacts** built from both.

This coupling makes ML harder in concrete ways:

- **You must version data, not just code** — and that's hard. How do you version a huge dataset? How do you know whether a sample is good or bad? Not all data is equal: if you have a million normal lung scans and a thousand cancerous ones, a cancerous scan is far more valuable — and indiscriminately ingesting data invites **data‑poisoning** attacks.
- **Model size.** Models with hundreds of millions to billions of parameters need gigabytes of RAM and are a serious engineering challenge to deploy — especially on **edge devices** (Chapter 7) — and to run *fast enough* to be useful (an autocomplete that lags behind your typing is worthless).
- **Monitoring and debugging** opaque models in production is genuinely hard; with little visibility into their reasoning, diagnosing failures — or even noticing them quickly — is nontrivial.

The encouraging news: these challenges are being solved rapidly. When BERT (340M parameters, 1.35 GB) appeared in 2018, it was deemed too big and slow to be practical. Two years later, BERT and its variants powered nearly every English Google search.

## Key takeaways

::: takeaways
- An ML system is **far more than the algorithm** — it spans business goals, data, infrastructure, interface, deployment, and monitoring. Design the *system*, not just the model.
- Reach for ML only when you can **learn complex patterns from existing data to predict on unseen data** that shares those patterns — and it especially pays off when the task is **repetitive, forgiving of errors, at scale, and constantly changing.**
- **Don't** use ML when it's unethical, when a simpler solution works, or when it isn't cost‑effective — but consider using ML for a *component* of a larger problem.
- Most ML *value and jobs* are in the **enterprise** and in **production**, not in consumer apps or research.
- Production flips research's priorities: **many stakeholders**, **low‑latency inference**, **messy shifting data**, and non‑negotiable **fairness** and **interpretability**.
- Always reason about latency as a **distribution** (p50/p90/p99), never as an average.
- ML couples **code + data + artifacts**, so you must test and version *data and models*, not just code.
:::

## Test your knowledge

```flashcards
Q: What are the five core conditions in the definition of machine learning?
A: ML **(1) learns (2) complex patterns** from **(3) existing data** to make **(4) predictions** on **(5) unseen data** that shares those patterns.
---
Q: Why is the *average* a poor way to summarize latency?
A: Latency is a distribution with a heavy tail. A single slow request (e.g., 3,000 ms) drags the mean far above what most users experience. Use percentiles (p50/p90/p99) instead.
---
Q: Give a problem that does **not** need ML, and say why.
A: Sorting listings into states by zip code — the pattern is simple and exact (each zip → one state), so a lookup table is better. ML is for *complex* patterns you can't write down.
---
Q: Name the three situations where you should *not* use ML.
A: When it would be unethical, when a simpler solution works, or when it isn't cost‑effective. (You can still use ML for a *sub‑component* of the problem.)
---
Q: How does ML break a core assumption of traditional software engineering?
A: SWE assumes code and data are separate. ML systems are part code, part data, and part artifacts — so you must test and version data and models, not just code.
---
Q: Latency vs. throughput — what's the difference, and how does batching change their relationship?
A: Latency is time per request; throughput is requests per unit time. Without batching, higher latency means lower throughput. With batching, higher latency can mean *higher* throughput (you process many queries together).
```
