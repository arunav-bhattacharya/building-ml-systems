Chapters 4–6 covered the *logic* of building ML systems; Chapters 7–9 covered deploying, monitoring, and updating them. All of that quietly assumed you had the tools and infrastructure to actually do the work. For many teams, that assumption is false. Data scientists routinely say they *know* the right thing to do — version that data, retrain on a schedule, run a shadow deployment — but **can't**, because their infrastructure won't let them.

The more complex a system, the more it benefits from good infrastructure. Set up well, infrastructure automates the boring parts, reduces the surface area for bugs, frees engineers from specialized plumbing, and unlocks new use cases. Set up badly, it is painful to use and expensive to replace. This chapter is about setting it up right.

A crucial caveat up front: **there is no one-size-fits-all infrastructure.** What you need depends on how many applications you run and how specialized they are. The diagram below shows the spectrum.

The storage layer has been almost entirely **commoditized** and moved to the cloud over the last decade. At its simplest it is a hard disk; in practice it is Amazon S3, Snowflake, or BigQuery — possibly spread across several of them (which makes engineers grumpy). Storage has become so cheap that most companies simply keep everything. Because Chapter 3 covered data systems in depth, this chapter concentrates on **compute**.

### The compute layer

The compute layer is all the compute resources a company can access, plus the mechanism that decides how they are used. The amount available sets the ceiling on how scalable your workloads can be. At its simplest it is one CPU or GPU core; most commonly it is cloud compute from AWS EC2 or GCP.

Compute is sliced into **compute units** that run concurrently. A CPU core might support two threads, each a unit running its own job; or several cores might be fused into one larger unit for a bigger job. A unit can be **ephemeral** — created for a single short-lived job (an AWS Step Function, a GCP Cloud Run) and destroyed when it finishes — or **more permanent**, not tied to any one job, like a virtual machine (often called an *instance*).

::: callout note "Units aren't always cores"
Some compute layers abstract cores away entirely. Spark and Ray use the **job** as their unit; Kubernetes uses the **pod** — a wrapper around one or more containers and its smallest deployable unit. You can put several containers in a pod, but you can't start or stop them independently.
:::

### How compute is measured

To run a job, you load its data into the unit's memory, then perform operations (add, multiply, convolve, …). If the unit lacks the memory to hold the data, the job is impossible without special out-of-memory handling. So a compute unit is characterized by **two metrics: how much memory it has, and how fast it runs operations.**

**Memory** is the easy one — measured in GB, and clearly more is better (and pricier). As of the book's writing, most ML workloads need 4–8 GB, and 16 GB covers nearly all of them. Some providers also advertise *memory bandwidth* or *I/O bandwidth*, because how fast you move data in and out of memory matters too.

**Operation speed** is contentious. The usual metric is **FLOPS** — floating-point operations per second — and vendors love to advertise teraFLOPS on their GPUs, TPUs, and IPUs. But FLOPS is slippery for two reasons:

- Vendors disagree on what counts as one *operation*. If hardware *fuses* two operations into one, is that one op or two?
- A unit *capable* of a trillion FLOPS won't actually run your job that fast. The ratio of FLOPS your job achieves to FLOPS the unit can deliver is its **utilization.** A job running 0.3M FLOPS on a 1M-FLOPS-capable unit has 30% utilization. 100% is essentially unreachable; depending on hardware and workload, even 50% may be good or bad. Utilization depends heavily on I/O bandwidth — you can't compute on data you haven't loaded yet.

::: callout key "Forget FLOPS — count cores and benchmark"
Because FLOPS is so hard to reason about, most people just look at the **number of cores and the memory** of an instance — e.g., "4 CPU cores, 8 GB." Two practical notes: AWS sells **vCPUs** (virtual CPUs), which for practical purposes are about *half* a physical core; and to compare real-world speed, use a benchmark like **MLPerf**, which reports how long hardware takes on standard jobs (training ResNet-50 on ImageNet, running BERT-large on SQuAD).
:::

How is compute billed? Cloud providers charge for the exact amount you use. **On-demand instances** are available whenever you ask, at full price; **spot instances** are surplus capacity sold at a discount but reclaimable at any moment — cheaper, but harder to rely on.

### Public Cloud Versus Private Data Centers

Like storage, compute is largely commoditized — so instead of running their own data centers, companies pay AWS, Azure, or GCP for exactly the compute they consume. The cloud's killer feature is **elasticity**, and it is especially well-suited to ML, whose workloads are **bursty**: teams hammer compute for a few weeks of experimentation during development, then settle into a steadier production load.

::: callout example "Why elasticity matters"
Suppose your workload needs 1,000 CPU cores on *one* day of the year and only 10 cores the rest. Build your own data center and you must buy 1,000 cores up front and let them idle 364 days. With the cloud, you pay for 1,000 cores that one day and 10 the rest — and most providers scale up and down **automatically**, cutting operational overhead to near zero.
:::

But the cloud is elastic, **not magical** — it has finite capacity. Providers cap how much compute you can use at once; some caps can be raised by petition, some can't. (At the book's writing, EC2's largest instance, X1e, topped out at 128 vCPUs and ~4 TB of memory.) And having lots of compute available doesn't make it *easy* to use — especially when you're juggling spot instances to save money.

Elasticity and ease of use have driven a clear shift: enterprise cloud spending grew 35% in 2020 to nearly \$130B, while data-center spending *fell* 6% to under \$90B. Yet there's a twist as companies scale up.

::: callout warning "The cloud's hidden cost at scale — and 'repatriation'"
Leveraging the cloud beats building your own early on, but it gets **less defensible as you grow.** The VC firm a16z found cloud spend runs to roughly **50% of cost of revenue** for public software companies, and estimated that ~\$100B of market value is being lost across the top 50 such firms *relative to running infrastructure themselves.* Dropbox saved **\$75M** over two years before its IPO largely by moving workloads off the public cloud — a move called **cloud repatriation.** Getting onto the cloud is easy; getting off is hard, requiring real up-front investment in hardware and engineering. Hence the common compromise: a **hybrid** approach, keeping most workloads on the cloud while slowly building out private data centers.
:::

::: callout note "Multicloud — usually not a choice"
Spreading workloads across multiple providers reduces dependence on any one and avoids **vendor lock-in**; a 2019 Gartner study found 81% of organizations use two or more public clouds (a common ML pattern is training on GCP/Azure, serving on AWS). But as reviewer Josh Wills put it, *"Nobody in their right mind intends to use multicloud."* It usually *happens* to you — different teams pick different clouds, an acquired company is already elsewhere, or a strategic investor (Microsoft, Google) nudges a portfolio company over. Moving data and orchestrating jobs across clouds is genuinely hard.
:::

## Development Environment

The **dev environment** is where ML engineers write code, run experiments, and interact with the production environment (deploying champion models, evaluating challengers). It has three parts: an **IDE**, **versioning** tools, and **CI/CD**.

Here's the counterintuitive part: outside a handful of tech companies, the dev environment is **severely underinvested in.** As Ville Tuulos observes, plenty of companies have polished production infrastructure yet solve "how is the code written, debugged, and tested in the first place" in an ad hoc way. His advice is blunt and worth tattooing on a wall:

::: callout key "If you fix only one thing, fix the dev environment"
*"If you have time to set up only one piece of infrastructure well, make it the development environment for data scientists."* Engineers spend most of their time here, so any improvement translates **directly** into productivity. It's the least abstract layer to a data scientist — and therefore the highest-leverage place to invest.
:::

### Dev Environment Setup

A good dev environment bundles every tool that makes the job easier, including tools for **versioning**. Today teams stitch together an ad hoc set: **Git** for code, **DVC** for data, **Weights & Biases** or **Comet.ml** for experiment tracking during development, and **MLflow** for tracking model artifacts at deployment. Versioning matters in all software, but *more* in ML because there's so much you can change — code, parameters, the data itself — and you must be able to reproduce a past run later. The dev environment should also include a **CI/CD test suite** (orchestrated by, e.g., GitHub Actions or CircleCI) to test code before it reaches staging or production.

The piece engineers touch most is the **IDE** — the editor where code is written. IDEs support many languages and come as native apps (VS Code, Vim) or browser-based tools (AWS Cloud9). But data scientists live just as much in **notebooks** — Jupyter, Google Colab (which even offers free GPUs). Notebooks are more than editors: they hold arbitrary artifacts (images, plots, tabular data), which makes them ideal for exploratory analysis and inspecting training results.

::: callout analogy "Notebooks are stateful — a blessing and a curse"
A notebook **retains state between runs**, like a kitchen where the half-cooked dish stays on the stove. If your program fails at step 4, you rerun *only* step 4 — you don't reload a giant dataset from scratch. That's a huge time-saver. The curse: you can run cells **out of order** (cell 4, then 2, then 3), so a notebook that "works" on your machine may be impossible for anyone else to reproduce — unless you document the intended run order. (Hence the running joke about un-reproducible notebooks.)
:::

Some companies put notebooks at the *center* of their data-science infrastructure. Netflix's "Beyond Interactive" post describes tooling to supercharge them — **Papermill** (run many parameterized notebooks concurrently and summarize their metrics) and **Commuter** (a hub to view, find, and share notebooks). The **nbdev** library goes further, encouraging you to write docs and tests right alongside the notebook code.

The arc went like this. First, unpinned packages (`torch` instead of `torch==1.10.0+cpu`) caused "works on my machine" bugs → **pin every version.** Then a concurrency bug appeared only on Python 3.8, not 3.9 → **standardize the runtime.** Then a teammate's new M1 MacBook broke tools like Docker that weren't yet M1-ready → the team gave up on local machines entirely and moved to a **cloud dev environment**, where everyone uses the same packages *on the same type of machine*, provided by a cloud vendor.

A cloud dev environment can pair with a **cloud IDE** (AWS Cloud9, or Amazon SageMaker Studio with hosted JupyterLab) — but the more popular pattern is a **local IDE connected to the cloud**, e.g., VS Code on your laptop reaching the cloud instance over SSH. Tools and packages should be standardized; **IDEs** are trickier, because engineers get emotionally attached (recall the decades-long Vim-vs-Emacs editor war). VS Code is a strong default given its smooth cloud-instance integration. The book's startup used **GitHub Codespaces**; an SSH-able EC2 or GCP instance works too.

The benefits of going cloud are substantial: IT supports one image instead of a thousand laptops; remote work is trivial (SSH in from anywhere); security improves (revoke a stolen laptop's access instantly); and — the biggest win for teams whose production is on the cloud — it **shrinks the gap between dev and prod.** Sometimes it's a necessity: when data can't legally leave the cloud, a hosted notebook reading from S3 may be the *only* way to touch it. The classic worry — forgetting to shut an instance down — is largely solved: Codespaces auto-stops after 30 minutes idle, and a 4-vCPU / 8-GB box runs ~\$0.10/hour (~\$73/month even if never stopped), cheap against the cost of engineering time. The trade-offs: some firms can't put code or data on the cloud at all, and there's setup cost plus the need to teach "cloud hygiene" (secure connections, compliance, not wasting spend).

### From Dev to Prod: Containers

During development you usually work on a fixed number of machines (often just one) because your workload is stable. **Production is different**: a service may spread across many instances, and that number changes with incoming traffic — a celebrity tweets about your app and traffic spikes 10×. Cloud providers handle the **autoscaling** (spinning instances up and down), but you still have to make each fresh instance *ready*: in production, instances are allocated dynamically, so the environment is inherently **stateless** — every new instance must install its dependencies from a predefined recipe.

That recipe problem is exactly what **containers** solve, and **Docker** is the dominant tool.

::: callout analogy "Dockerfile → image → container: a mold and its castings"
A **Dockerfile** is a step-by-step recipe: install these packages, download this pretrained model, set these env vars, `cd` here. Run that recipe and you get a **Docker image** — think of it as a *mold*. From one mold you can stamp out many identical **containers** — the running castings. Build an image from scratch, or from a base (e.g., an NVIDIA image with TensorFlow pre-tuned for GPUs) and layer your app on top. A **container registry** (Docker Hub, AWS ECR) is where images are shared publicly or within an org — like a library of molds.
:::

Real applications usually need **more than one container**. Two reasons recur. First, **different resource profiles**: featurizing code may be fast but memory-hungry, while training is slow but memory-light — run featurizing on cheap CPU instances and training on GPU instances, in separate containers, rather than paying for high-memory GPUs. Second, **conflicting dependencies**: if your featurizer needs NumPy 0.8 but your model needs NumPy 1.0, they can't share a container.

Run a hundred microservices and you may have a hundred containers to build, start, allocate, and stop — a miserable manual chore. **Container orchestration** automates it. **Docker Compose** handles multiple containers on a *single host*; once containers span *many hosts*, you reach for **Kubernetes (K8s)**. K8s networks containers so they can communicate and share resources, spins instances up when you need more compute and down when you don't, and maintains high availability. It was one of the fastest-growing technologies of the 2010s and is now ubiquitous — but it is famously *not* data-scientist-friendly, which has driven a whole movement to keep data-science workloads away from raw K8s (more on that next).

## Resource Management

In the pre-cloud world (and still in private data centers), storage and compute were **finite**, so resource management was about squeezing maximum utilization from scarce resources — giving more to one app meant taking from another, and the elaborate logic to optimize that cost real engineering time. The cloud flips the framing: resources are **elastic**, so adding to one app no longer starves another, and the question shifts from *"how do we maximize utilization?"* to *"how do we use resources cost-effectively?"*

::: callout key "When engineer-time costs more than compute-time"
In most of the world, an engineer's time is worth more than a machine's. So it often makes sense to **automate a workload even if that wastes some compute**, because it frees engineers for higher-return work. Put differently: when a problem can be solved by throwing *more compute* at it or by *more engineering time*, the compute solution is usually preferred.
:::

### Cron, Schedulers, and Orchestrators

Two traits of ML workflows shape how you manage their resources: they are **repetitive** (retrain weekly, score every four hours) and they have **dependencies** between steps. A tooling ladder addresses these in increasing sophistication.

**Cron** sits at the bottom: it runs a script at a predetermined time and tells you whether it succeeded or failed. That's *all* it does. Cron knows nothing about dependencies — you can run job A then job B on fixed schedules, but you can't express "run C if A succeeds, run D if A fails."

Dependencies are the second trait, and they get complex. Consider a workflow: (1) pull last week's data, (2) extract features, (3) train models A and B, (4) compare them on a test set, (5) deploy the winner. Each step depends on the previous one, and step 5 is a **conditional dependency** — the action depends on the *outcome* of step 4. Such workflows form a **DAG (directed acyclic graph)**: *directed* to encode dependency order, *acyclic* because a cycle would loop forever. Nearly every workflow tool asks you to express your pipeline as a DAG.

::: callout note "Schedulers vs. orchestrators — *when/what* vs. *where*"
A **scheduler** is "cron that understands dependencies." It takes a DAG (or an **event-based trigger** — "start when event X happens"), decides *when* each step runs, handles **retries** on failure, and uses **queues** to prioritize and allocate jobs — so it must know what resources each job needs (specified by you or estimated; Google's **Borg** even reclaims over-requested resources). Examples: **Slurm**, Borg.

An **orchestrator** answers a different question — *where* do the resources come from? It works at lower-level abstractions (machines, instances, clusters, replication) and, when there are more jobs than instances, **provisions** more. The dominant orchestrator is **Kubernetes**; most teams use it as a hosted service (AWS **EKS**, Google **GKE**) rather than suffer setting up their own cluster.
:::

### Data Science Workflow Management

Where do the *data-science-specific* tools — Airflow, Argo, Prefect, Kubeflow, Metaflow — fit in this scheduler-vs-orchestrator picture? At heart, **workflow management tools manage workflows**: you define a pipeline as a **DAG** of steps (each step is a **task**) using either **code (Python)** or **configuration files (YAML)**. Almost all ship a scheduler, so think of them as schedulers focused on the *whole workflow* rather than individual jobs; once defined, the workflow's scheduler works with an orchestrator to actually allocate resources. The book compares five tools; the table captures the essence.

| Tool | Origin / year | Workflow defined in | Containers | Parameterized & dynamic? | Key trait |
|---|---|---|---|---|---|
| **Airflow** | Airbnb, 2014 | **Code** (Python) | Possible via `DockerOperator`, but awkward (monolithic) | **No** — DAGs are static & not parameterized | Huge operator library; "configuration as code" pioneer, but oldest and most dated |
| **Prefect** | by an Airflow core contributor | **Code** (Python) | Possible, but not first-class (still write Dockerfiles) | **Yes** | A direct, parameterized, dynamic answer to Airflow |
| **Argo** | — | **Config** (YAML) | **Every step in its own container** | **Yes** | Container-native; but messy YAML and **K8s-only** (need minikube to test locally) |
| **Kubeflow** | — | Python (+ Dockerfile + YAML) | Yes (built on Argo) | **Yes** | Run the *same code* in dev & prod; abstracts other tools' boilerplate by adding its own |
| **Metaflow** | — | **Code** (Python) | Auto-generated from `@conda` | **Yes** | Best UX; seamless dev↔prod via `@batch`; per-step environments |

Three observations the book emphasizes. **Airflow** champions "configuration as code" but, having no predecessor to learn from, suffers from being *monolithic* (the whole workflow in one container), having *non-parameterized* DAGs (different learning rates → different workflows), and *static* DAGs (it can't create steps at runtime — e.g., one step per unknown-count database record). **Argo** fixes containers (each step is isolated) but locks you to K8s and buries you in YAML. **Kubeflow and Metaflow** both aim to run the *same workflow in dev and prod*, giving data scientists the prod environment's full compute from a local notebook.

::: callout example "Metaflow's developer experience"
Metaflow lets you decorate a step with `@conda(libraries={...})` to declare its libraries and resources, and it auto-builds the container — no hand-written Dockerfile or YAML. Crucially, you can mix environments **within one workflow**: a light step runs locally, and adding `@batch(gpu=2, memory=16000)` to the next step ships *that* step to AWS Batch. So you prototype on a small local dataset, then scale to the cloud by adding a decorator — even running two ensemble models, one local and one on AWS, side by side. (Kubeflow, by contrast, still makes you write Dockerfiles and YAML per component — it "abstracts away other tools' boilerplate by making you write Kubeflow boilerplate.")
:::

## ML Platform

As a company finds ML uses across more and more applications, a pattern emerges: the same supporting tools — feature management, model management, monitoring — get rebuilt for each app. The fix is to **share one set of tools** across applications. That shared toolset is the **ML platform**, and dedicated ML-platform teams have become common since early 2020 (often spun out of whichever team — frequently the recommender team — had the most mature tooling).

Because the concept is new, *what* an ML platform contains varies by company. The book focuses on the three components it sees most: **model deployment**, the **model store**, and the **feature store** (monitoring was covered in Chapter 8). When evaluating any tool in these categories, two cross-cutting questions apply:

::: callout note "Two questions for any ML-platform tool"
1. **Does it work with your compute layer?** You serve models from your cloud/data center, and tools typically integrate with only a handful of providers. Nobody wants to adopt a *new cloud* just to use one tool.
2. **Open source or managed service?** **Open source** you can self-host — better for data security, but extra engineering time to maintain. A **managed service** is less work, but your models (and some data) live on the vendor's service, which may violate regulations. Some managed services run inside your **virtual private cloud** to help with compliance.
:::

### Model Deployment

Once a model is trained and tested, you make its predictions accessible to users. From Chapter 7, the simplest deployment pushes the model and its dependencies somewhere production-accessible, then exposes it as an **endpoint**: for **online prediction** the endpoint triggers the model to predict; for **batch prediction** it fetches a precomputed result.

A **deployment service** automates both the pushing and the endpoint-exposing. Deployment is the **most mature** ML-platform component — *"deploying is the name of the game"* — so tools abound. Every major cloud offers one (AWS **SageMaker**, GCP **Vertex AI**, Azure **Azure ML**, Alibaba's Machine Learning Studio), alongside startups (**MLflow Models**, **Seldon**, **Cortex**, **Ray Serve**). When choosing, check how easily it does **both** online *and* batch prediction — online at small scale is usually trivial (hit an endpoint, get a prediction back), but **batch is trickier** (set up batch jobs, store the outputs), so many companies run separate pipelines (e.g., Seldon for online, Databricks for batch). Also confirm it makes your preferred **test-in-production** techniques from Chapter 9 (shadow deployment, canary release, A/B testing) easy — ensuring a model's quality *before* it goes live is still an open problem.

### Model Store

Model stores sound trivial — "they store models, so just upload to S3" — and that dismissiveness is exactly the trap.

::: callout warning "Why 'just upload the model to S3' fails"
A model's performance drops for some inputs. A DevOps engineer is paged, traces it to a model, and needs to alert *the* data scientist who built it — but which of 20 data scientists? Say the right one is found. She reruns the original notebook locally and gets **different outputs than production.** Why? Maybe the wrong binary was uploaded; maybe the feature list is stale; maybe the featurization code is outdated; maybe the data pipeline differs. Without knowing which, she can't fix it — and this *assumes* she still has the notebook and hasn't quit or gone on vacation. Storing the model binary alone is nowhere near enough.
:::

To support debugging and maintenance, a model store should track far more than the model file. The book lists **eight types of artifacts** (many of which also belong in the *model card* from Chapter 11):

| # | Artifact | What it captures |
|---|---|---|
| 1 | **Model definition** | The model's *shape* — loss function, number of layers, parameters per layer |
| 2 | **Model parameters** | The actual parameter *values*; combined with the definition, they recreate a usable model |
| 3 | **Featurize & predict functions** | How to turn a request into features and into a prediction (usually wrapped in the endpoint) |
| 4 | **Dependencies** | Python version, packages — typically bundled into a container |
| 5 | **Data** | Pointers to (or name/version of) the training data; e.g., a DVC commit |
| 6 | **Model generation code** | Frameworks used, how it was trained, train/valid/test splits, # experiments, hyperparameter ranges, final hyperparameters |
| 7 | **Experiment artifacts** | Outputs from development — loss curves, test-set numbers |
| 8 | **Tags** | Metadata for discovery/filtering — owner, task (e.g., "fraud detection") |

In reality, companies store only a *subset*, and often **scattered**: definitions and parameters in S3, dependency containers in ECS, data in Snowflake, experiment artifacts in Weights & Biases, featurize/predict functions in Lambda — with the locations tracked, if at all, in a README that's easily lost. A truly general model store remains **unsolved**: **MLflow** is the most popular non-cloud option, yet three of its six top Stack Overflow questions are about storing and accessing artifacts. Lacking a good option, companies like **Stitch Fix** build their own, tracking the serialized model, its Python environment, the Git commit of the generation code, tags, and more.

### Feature Store

"Feature store" is a loaded term that means different things to different people. At its core, a feature store tackles **three problems** — and a given product may address one, two, or all three.

::: callout key "The three jobs of a feature store"
1. **Feature management** — a catalog. With many models each using many features (Uber had ~10,000 features across teams in 2017), features useful to one model often help another. A feature store lets teams **share and discover** features and manage access/roles (e.g., restricting sensitive financial features). Tools: **Amundsen** (Lyft), **DataHub** (LinkedIn).
2. **Feature computation** — feature *logic* ("average meal-prep time from yesterday") must actually be **computed** over your data. If a shared feature is cheap, compute it on demand; if expensive, compute once and **store** the result. Here the feature store acts like a data warehouse.
3. **Feature consistency** — the big one (next).
:::

::: callout analogy "Writing the same feature twice — the consistency trap"
In development you define features in Python; production code is often rewritten in Java or C for speed. So the *same* feature gets written **twice** — once for training, once for inference. This is tedious *and* dangerous: if the two versions drift even slightly, the model behaves strangely and the bug is brutal to find. It's like a recipe translated into two languages by two cooks — subtle differences creep in, and the dish comes out wrong only sometimes. A modern feature store's key selling point is to **unify batch and streaming feature logic**, guaranteeing the features used in training match those used at serving time.
:::

Feature stores took off around 2020, and exact capabilities vary: some only manage definitions, some also compute, some also do **feature validation** (flagging features that violate a schema) while others leave that to monitoring. The most popular open-source option is **Feast** (strong at batch, weaker at streaming); **Tecton** is a fully managed store promising both batch and online features, though adoption is slowed by the deep integration it requires; SageMaker and Databricks ship their own takes. In a January 2022 survey of 95 companies, only ~40% used a feature store at all — and of those, **half had built their own.**

## Build Versus Buy

How much to invest in infrastructure depends not just on your applications and scale, but on **what you build in-house versus buy.** The choice has real staffing consequences: using fully managed Databricks might need one engineer, while self-hosting your own Spark EMR clusters might need five more.

The decision lives on a spectrum. At one extreme, **outsource everything** to a vendor that delivers ML applications end-to-end — your only infrastructure is moving data to the vendor and predictions back. At the other extreme, **build everything in-house** (even your own data centers), as a company handling data too sensitive for any third party must. Most companies are **in between** — compute on EC2, warehouse on Snowflake, but their *own* feature store and monitoring dashboards.

The book highlights **three factors** that heads of infrastructure weigh:

- **Stage of the company.** Early on, lean on vendors to launch fast and conserve scarce resources for your core product. As use cases multiply, vendor costs can balloon until building your own becomes cheaper.
- **Focus / competitive advantage.** Stitch Fix's Stefan Krawczyk: *"If it's something we want to be really good at, we'll manage that in-house. If not, we'll use a vendor."* Non-tech firms (retail, banking, manufacturing) bias toward **buying** — even point solutions like a demand-forecasting service — because ML infrastructure isn't their focus. Tech firms bias toward **building**, preferring modular, customizable services they can plug-and-play.
- **Maturity of available tools.** Sometimes you'd *prefer* to buy but no vendor is good enough, so you build (often atop open source). This is why early adopters — big tech — built custom infrastructure: nothing mature existed. That in turn makes every company's stack different, so later vendors struggle to sell into big tech's bespoke setups (founders are warned to avoid "**integration hell**" and target clean-slate startups instead).

::: callout warning "Building is not automatically cheaper"
A common myth is that building beats buying on cost. Not necessarily: building means hiring more engineers to build *and maintain* the infrastructure, and it carries a **cost of innovation** — custom, in-house systems make adopting new technology hard because of integration friction. As ex-Better.com CTO Erik Bernhardsson noted, vendor/product selection is one of a CTO's most important jobs, and it only grows in importance as the infrastructure space expands.
:::

## Summary

Bringing ML to production is, fundamentally, an **infrastructure problem.** This chapter walked the four layers that let data scientists develop and deploy reliably. The **storage and compute** layer is the heavily-commoditized foundation: most companies rent it from the cloud, paying for exactly what they use — though costs grow prohibitive at scale, pushing some large companies toward **cloud repatriation** and hybrid setups. The **development environment** is where engineers spend most of their time, so improving it (starting with **standardization**, all the way to **containers**) pays off directly in productivity. **Resource management** evolved from **cron** to **schedulers** (when/what) to **orchestrators** (where), with data-science workflow tools — Airflow, Argo, Prefect, Kubeflow, Metaflow — sitting on top. The **ML platform** — **deployment**, **model store**, and **feature store** — is an emerging shared toolset, with the feature store's training/serving **consistency** guarantee a standout feature. And underlying every component is the inescapable, context-dependent **build-vs-buy** question that quietly consumes engineering managers and CTOs.

## Key takeaways

::: takeaways
- ML infrastructure has **four layers** — storage & compute, resource management, the ML platform, and the dev environment — and there's **no one-size-fits-all**; most companies operate at "reasonable scale" (GB–TB/day), between ad hoc analytics and Google-scale custom builds.
- Compute is measured by **memory** and **operation speed**; FLOPS is slippery (definition disputes, real **utilization** ≪ peak), so in practice count **cores + memory** and use benchmarks like MLPerf.
- The **cloud** wins early via **elasticity** and pay-per-use, perfect for bursty ML workloads — but cloud spend can hit ~50% of cost of revenue at scale, motivating **repatriation**, **hybrid**, and (usually involuntary) **multicloud**.
- The **dev environment** is the highest-leverage thing to get right; **standardize** tools, versions, and runtime — ideally with **cloud dev environments** and **containers** (Dockerfile → image → registry → Kubernetes) to kill "works on my machine."
- **Cron** runs on a schedule; **schedulers** add dependencies, triggers, retries, and queues; **orchestrators** provision the underlying machines. Workflow tools express pipelines as **DAGs** in code or YAML and trade off container support, parameterization, and dev↔prod portability.
- An **ML platform** centers on **deployment** (online + batch endpoints), a **model store** (eight artifact types, not just a binary — for lineage and debugging), and a **feature store** (management, computation, and **train/serve consistency**).
- **Build-vs-buy** hinges on your **stage**, whether the capability is a **competitive advantage**, and **tool maturity** — and building is *not* automatically cheaper once you count maintenance and the cost of innovation.
:::

## Test your knowledge

```flashcards
Q: Name the four layers of ML infrastructure described in this chapter.
A: **(1) Storage & compute** (the foundation), **(2) resource management** (schedulers/orchestrators), **(3) the ML platform** (deployment, model store, feature store), and **(4) the dev environment** (where code is written and experiments run).
---
Q: Why is FLOPS a poor way to compare compute units, and what do people use instead?
A: Vendors disagree on what counts as an "operation," and a unit's *achieved* FLOPS (its **utilization**) is far below its peak. In practice people count **cores + memory** and rely on benchmarks like **MLPerf**.
---
Q: What is "cloud repatriation," and why does it happen?
A: Moving workloads from the public cloud **back** to private data centers. The cloud is cheap to start with but its cost grows prohibitive at scale (a16z: ~50% of cost of revenue), so large companies (e.g., Dropbox, which saved \$75M) repatriate — usually ending up **hybrid**.
---
Q: Distinguish cron, a scheduler, and an orchestrator.
A: **Cron** just runs a script at a set time (no dependencies). A **scheduler** adds dependency-aware DAGs, event triggers, retries, and resource-aware queues — it decides *when/what*. An **orchestrator** manages machines and *provisions* more when needed — it decides *where*.
---
Q: A model store should hold much more than a model binary. Name several of the eight artifacts.
A: Model definition, model parameters, featurize & predict functions, dependencies, training data (pointers/version), model generation code, experiment artifacts, and tags — needed for lineage, debugging, and maintenance.
---
Q: What is the "feature consistency" problem, and how does a feature store solve it?
A: Features defined in Python for training are often rewritten (e.g., in Java/C) for serving, so the two can drift and cause subtle bugs. A feature store **unifies batch and streaming feature logic**, guaranteeing training-time and serving-time features match.
---
Q: In Docker, what's the relationship between a Dockerfile, an image, and a container?
A: A **Dockerfile** is the recipe; running it produces a **Docker image** (a reusable "mold"); running the image produces a **container** (a live instance). One image → many identical containers. Images are shared via a **container registry** (Docker Hub, ECR).
```

```quiz
Why is the cloud especially well-suited to ML workloads specifically?
- ( ) The cloud offers truly infinite compute
- (x) ML workloads are bursty — heavy during experimentation, steadier in production — so elastic pay-per-use fits well
- ( ) The cloud is always cheaper than a data center at any scale
- ( ) ML models can only run on cloud GPUs
> Data-science workloads surge during weeks of experimentation, then settle down. Elastic, pay-for-what-you-use compute matches that burstiness — but the cloud is finite and gets costlier at scale.

A workflow must "deploy model A if it beats model B, otherwise deploy B." Plain cron cannot express this because…
- ( ) cron can't run scripts on a schedule
- ( ) cron requires Kubernetes
- (x) cron has no notion of dependencies or conditional outcomes between jobs
- ( ) cron can only run one job ever
> Cron only runs a script at a set time and reports success/failure. Conditional, dependency-aware logic requires a scheduler, which understands DAGs and outcomes.

Which statement best captures Airflow's main limitations versus newer tools?
- ( ) It can't define workflows in Python
- (x) Its DAGs are monolithic, non-parameterized, and static (no new steps created at runtime)
- ( ) It has no scheduler at all
- ( ) It only runs on a single laptop
> As the earliest of the tools, Airflow packages workflows into one container, can't pass parameters into DAGs, and can't generate steps dynamically at runtime — gaps that Prefect, Argo, Kubeflow, and Metaflow address.

According to the chapter, which factor does NOT typically push a company toward *building* infrastructure in-house?
- ( ) The capability is a core competitive advantage
- ( ) No vendor tool is mature enough for their needs
- ( ) They have grown large enough that vendor costs exceed build costs
- (x) They are a small non-tech company wanting to launch as fast as possible
> Early-stage and non-tech companies bias toward **buying** to move fast and focus on their core product. Building is favored by scale, competitive-advantage focus, and a lack of mature tools.

What is the primary value of a *model store* beyond blob storage?
- ( ) It compresses models to run on edge devices
- (x) It tracks the artifacts, metadata, and lineage needed to reproduce and debug a model in production
- ( ) It automatically retrains models on a schedule
- ( ) It replaces the need for a feature store
> Storing just the binary leaves you unable to explain why production and local outputs differ. A model store versions definitions, parameters, code, data pointers, dependencies, experiments, and tags for lineage and debugging.
```
