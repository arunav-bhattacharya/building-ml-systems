Every chapter so far has been about the *technical* craft of building ML systems — data, features, training, deployment, monitoring. But an ML system never lives in a vacuum. It is **built by humans, used by humans, and leaves marks on society.** The business decision-makers who fund it, the users who click on its predictions, and the engineers who keep it alive are all part of the system too.

This final chapter steps back from the pipeline to look at three human dimensions. First, how ML's quirky nature — **probabilistic, mostly-correct, and sometimes slow** — reshapes *user experience*. Second, how to organize the *people* on an ML project so a diverse cast collaborates well. And third — the topic the author calls the most important in the whole book — **responsible AI**: making sure the systems we ship do more good than harm.

::: callout note "Why ML breaks the user's mental model"
Traditional software is **deterministic**: same input, same output, every time. An ML system is **probabilistic** — run it twice on the *exact same* input and you may get *different* results. Worse, its predictions are only *mostly* correct, and you usually can't tell *which* inputs it will get wrong. And because models can be huge, a prediction may arrive late. Users raised on deterministic software find all three surprising.
:::

- The three ways ML's probabilistic, mostly-correct, slow nature degrades **user experience** — and how to fix each

- Why **subject-matter experts** belong in the whole lifecycle, not just labeling
- The debate over **end-to-end data scientists** vs. siloed teams — and why good tooling is the deciding factor

- A practical **framework** for building responsible AI: find bias sources, know data's limits, weigh trade-offs, act early, and use model cards

### Ensuring User Experience Consistency

Users expect software to behave *consistently*. If Chrome moved its "minimize" button from the top-left to the right overnight, you would be confused and annoyed — even though nothing about the button's *function* changed. **Consistency itself is part of a good experience.**

ML predictions are probabilistic and therefore *inconsistent*: what the model suggests to a user today may differ from what it suggests tomorrow, depending on context. For features meant to *help* users, that flicker can be a hindrance rather than a help.

::: callout example "Booking.com's auto-suggested filters"
Booking.com lets you narrow accommodations with roughly **200 filters** ("breakfast included," "pet friendly," "non-smoking"). There are so many that finding the right ones is tedious, so the applied-ML team built a model to **suggest filters** based on what you'd used earlier in the session. The catch: if the model kept proposing *different* filters each visit, users got confused — especially when a filter they'd *already applied* seemed to vanish. The fix was a **rule layer**: define when the system *must* return the same recommendations (e.g., once a user has applied a filter) versus when it's free to suggest new ones (e.g., when they change destination).
:::

This is the **consistency–accuracy trade-off**: the recommendations the model scores as *most accurate* are not always the ones that give the user a *consistent, trustworthy* experience. Sometimes you deliberately serve a slightly-less-"optimal" but stable result, because consistency is what the user actually values.

### Combatting "Mostly Correct" Predictions

The previous section wanted *more* consistency. This one wants the opposite — *more diversity* — because of a different problem.

Large language models like GPT-3 can produce useful output for an enormous range of tasks with little or no task-specific training. Feed one a description of a web page and it can emit the **React code** to build it. The trouble: these predictions are only *mostly correct*, and fine-tuning them to be reliably correct is expensive. A "mostly correct" answer is great **when the user can easily fix it** — a customer-support agent can quickly edit a draft reply, which is faster than writing from scratch.

But "mostly correct" is nearly useless **when the user *can't* fix it.** The generated React might not run, or might render the wrong page. A React engineer could patch it in seconds — but the people drawn to a "describe-a-page-and-get-code" tool are precisely those who *don't know React*. That's why they wanted the tool in the first place.

::: callout key "Raise the odds with multiple candidates"
The fix for unfixable "mostly correct" output is to **show several predictions for the same input**, rendered so a *non-expert* can judge them. For the React example: have the model emit *multiple* code snippets, render each into a *visual* web page, and let the user pick the one that looks right. You don't need any single prediction to be reliable — you just need *one of several* to be good, and a human to recognize it.
:::

This pattern — humans selecting from or refining machine output — is **human-in-the-loop AI**. (The author recommends Jessy Lin's "Rethinking Human-AI Interaction" for going deeper.)

### Smooth Failing

Even a model that is normally fast can be slow on *certain* queries. This is common for models over **sequential data** — language or time-series models that take longer on long inputs than short ones. We've already covered inference latency (Chapter 1) and model compression for speed (Chapter 7), but compression only goes so far. What do you do about the *individual* queries that still take too long?

The answer is a **backup system**: something less capable than the main model but *guaranteed* to answer fast — a heuristic, a simple model, or even **cached pre-computed predictions**. A simple rule routes around the slowness: *if the main model takes longer than X ms, use the backup instead.* A fancier version trains *another* model to **predict how long** the main model will take for a given query and routes accordingly — though that predictor adds its own latency.

::: callout warning "A confusing experience is a failed experience"
Notice the through-line across all three UX problems: a technically *better* prediction can still produce a *worse* product. An inconsistent suggestion, an unfixable draft, or a laggy answer each erodes trust even if the model's offline metrics look great. Always evaluate predictions through the lens of *what the user can actually do with them.*
:::

## Team Structure

An ML project is staffed by more than data scientists and ML engineers: DevOps and platform engineers keep it running, and **subject-matter experts (SMEs)** supply domain knowledge. With such a mix, what's the best way to organize people? Two questions dominate: how to get cross-functional collaboration right, and the much-debated role of the **end-to-end data scientist.**

### Cross-functional Teams Collaboration

SMEs — doctors, lawyers, bankers, farmers, stylists — are routinely overlooked when ML systems are designed, yet many systems simply *wouldn't work* without their expertise. They are not just *users* of ML systems; they are effectively *developers* of them.

Most teams remember SMEs only at the **labeling** stage — you need a trained radiologist to mark whether a lung CT shows cancer. But as training becomes a continuous process in production, labeling and **relabeling** become ongoing too, spanning the whole project. SMEs add value far beyond labels: in **problem formulation, feature engineering, error analysis, model evaluation, reranking predictions, and UI design** — deciding how results should be shown to users or passed to other parts of the system.

::: callout note "The 'get your doctor to use Git' problem"
Mixing profiles creates friction. How do you explain an algorithm's limits to an SME with no statistics background? And ML demands that *everything* be **versioned** — but how do you translate "if there's a small dot in this region between X and Y, it might be cancer" into versioned code? *Good luck getting your doctor to use Git.* The pragmatic answer is to involve SMEs **early**, in the planning phase, and to **empower** them to contribute without filing a ticket every time. Many companies now build **no-code / low-code platforms** so SMEs can act directly — today mostly at labeling, QA, and feedback, but increasingly at dataset creation and issue investigation too.
:::

### End-to-End Data Scientists

A recurring theme of this book is that **ML production is an infrastructure problem as much as an ML problem.** Doing MLOps well needs *Ops* expertise — deployment, containerization, job orchestration, workflow management — on top of modeling. How do you get *both* skill sets onto a project? Companies pick one of two approaches.

**Approach 1 — a separate team productionizes the models.** The data-science team builds in dev; an Ops/platform/ML-engineering team ships to prod. **Hiring is easier** (one skill set per role) and each person has a single concern. But the drawbacks pile up:

| Drawback | What goes wrong |
|---|---|
| **Communication overhead** | One team blocks another. ("What one programmer does in one month, two do in two." — Fred Brooks) |
| **Debugging is hard** | When something breaks, you can't tell whose code — or *whose company's* code — caused it; fixing it needs multiple teams. |
| **Finger-pointing** | Even once the cause is known, each team may insist it's someone else's job to fix. |
| **Narrow context** | Nobody sees the whole pipeline. The platform team has ideas but can only react to requests; data scientists don't feel infra pain, so they don't push improvements. |

**Approach 2 — data scientists own the whole process.** Here the data scientist becomes a "grumpy unicorn" expected to know *everything* — querying, modeling, distributed training, endpoints, even Kubernetes and Airflow — and may end up writing more boilerplate than data science. The author once tweeted exactly such a sprawling skills list, and it resonated; Eugene Yan and Stitch Fix's Eric Colson have argued the same case for the **full-stack data-science generalist.**

::: callout key "The author's verdict: empower, don't overload"
Having learned low-level infrastructure herself, the author concluded it's **unreasonable** to expect data scientists to master it — *"expecting data scientists to know infrastructure is like expecting app developers to know how Linux kernels work"* (Erik Bernhardsson). People join ML to work with **data**, not to debug YAML and spin up AWS instances. The resolution isn't to abandon end-to-end ownership — it's to **give data scientists good tools.** With the right abstraction ("here's my data in S3, here are my steps, here's where to run them"), a platform can hide containerization, distributed processing, and automatic failover. At Netflix, the **specialists first build tools that automate their own parts**, and data scientists then leverage those tools to own projects end-to-end. The deciding factor between the two approaches is **infrastructure quality** — the subject of Chapter 10.
:::

## Responsible AI

> **Note:** Making intelligent systems responsible matters for *all* AI, not just ML — so this section says **AI** (the broader term that includes ML). It was written with contributions from Abhishek Gupta of the Montreal AI Ethics Institute.

**Responsible AI** is the practice of designing, developing, and deploying AI systems with **good intention and enough awareness** to empower users, earn trust, and ensure a fair, positive impact on society. Its pillars are **fairness, privacy, transparency, and accountability.**

These are no longer philosophical musings — they are urgent, practical concerns. As ML spreads into nearly every part of life, failing to make systems fair and ethical can cause **catastrophic** harm (the book *Weapons of Math Destruction* catalogs many). As a developer, you are responsible not only for how your system affects users, but for helping *every* stakeholder concretely implement ethics, safety, and inclusivity. We start with two public failures, then build a practical framework.

::: callout note "Want to go deeper?"
Responsible AI deserves its own books; this is a brief overview. The author points to **NIST SP 1270** (identifying and managing AI bias), the **ACM FAccT** conference, **Trustworthy ML**'s reading list, Sara Hooker's slide deck on fairness/security/governance, and the Gebru–Denton tutorials on fairness, accountability, transparency, and ethics. The **AI Incident Database** logs real-world failures — and remember, the public ones are only the tip of the iceberg.
:::

### Irresponsible AI: Case Studies

#### Case I: The automated grader's biases

In summer 2020 the UK canceled A-level exams (which determine college placement) because of COVID-19. The regulator, **Ofqual**, deployed an algorithm to assign final grades *without* a test, combining each student's prior attainment with teacher assessment. Teacher-only grading had been rejected as unfair across schools and prone to inflation; a statistical model was deemed "fairer." The published grades were anything but — they sparked public protests.

At first glance the problem looks like **low accuracy**: tested on 2019 data, the model was ~60% accurate, meaning 40% of grades would differ from reality. Ofqual countered that human examiners *also* agree only ~60% of the time, exposing the real uncertainty of judging a student at one moment. But as this book has stressed, **coarse accuracy is nowhere near enough** to evaluate a model that shapes thousands of futures. A closer look reveals **three failures.**

**Failure 1 — the wrong objective.** You'd expect the goal to be *grading accuracy for students.* Instead Ofqual optimized for **"maintaining standards" across schools** — fitting predicted grades to each school's *historical* distribution. If school A had out-performed school B in the past, the model was built to keep giving A's students higher grades. Ofqual prioritized fairness *between schools* over fairness *between students* — ignoring that richer schools out-perform poorer ones. The result: high-achieving students from historically low-performing schools were **downgraded** — an A pulled down to a B or C just because past cohorts at that school had scored Ds.

**Failure 2 — no fine-grained evaluation.** That school bias was just one of many. The model fed in teacher assessments without correcting for teachers' *inconsistency* across demographic groups, and it didn't account for compounding disadvantages faced by protected groups. Because it leaned on each school's history, Ofqual admitted it lacked data for **small schools** — and so used teacher grades alone for them, which handed *better* grades to **private-school** students (who tend to have smaller classes). **Slice-based evaluation** across school sizes and student backgrounds could have caught all of this *before* release.

**Failure 3 — no transparency.** Ofqual didn't reveal the system's *objective* until grades were published, so the public couldn't object while it was being built. It didn't tell teachers how their assessments would be used until *after* they'd submitted them (to stop gaming), and withheld the model itself until results day. Well-intentioned, perhaps — but the secrecy meant **no independent expert scrutiny.** The Royal Statistical Society questioned the model's legitimacy on exactly these grounds. *Any system operating on public trust must be reviewable by independent experts the public trusts.*

::: callout warning "Some decisions maybe shouldn't be automated at all"
Even fixing all three failures might not have placated the public — and that's the deeper lesson. The A-level case sits on the **murky boundary** between what algorithms *should* and *should not* decide. One can reasonably argue that, given the stakes, A-level grading should *never* have been automated. Until that boundary is drawn more clearly, misuse will keep recurring.
:::

#### Case II: The danger of "anonymized" data

Here the *algorithm* isn't the villain — the **data collection and interface** are. ML needs data: researchers need datasets to advance techniques, companies need data to find new use cases. But collecting and sharing data can violate the **privacy** of the people in it. The usual safeguard is to **anonymize** personally identifiable information (PII) — name, address, phone number, anything that lets an individual be "reasonably inferred." The problem: **anonymization is often not enough.**

::: callout example "Strava's fitness heatmap exposed military bases"
In 2018, fitness tracker **Strava** published a global heatmap of users' running, jogging, and swimming routes — aggregated from a *billion* activities over 27 billion km, and stated to be **anonymized**. But many users were military personnel, and their aggregated paths **traced out secret sites**: forward operating bases in Afghanistan, Turkish patrols in Syria, a likely guard patrol in a Russian operating area. Analysts suggested individual users' *names and heart rates* could even be recovered. The data was "anonymized" — and still leaked national-security secrets.
:::

Where did it go wrong? Strava's privacy default was **opt-out** — users had to *manually* disable tracking, and the settings were unclear and could only be changed on the website, not in the app. The lesson: **opt-in should be the default** (don't collect by default), and you must **educate users** about privacy controls. When the story broke, blame was shifted onto users ("don't use GPS devices on base"), but that only addresses the surface. The deeper truth is that our devices **constantly collect and transmit data** that must be stored somewhere — creating endless chances for interception and misuse. Strava's dataset is tiny next to Amazon's or Facebook's; a similar slip at that scale could be far more dangerous. **Developers must make the safe setting the default, even at the cost of collecting less data**, because users often lack the know-how to protect themselves.

#### 1. Discover sources of model bias

Bias can creep in at **every** stage of the lifecycle — that's exactly why it's so hard to fight. Audit each step:

| Stage | Question to ask |
|---|---|
| **Training data** | Is it representative of real-world data? Under-represented groups get worse predictions. |
| **Labeling** | How is label quality measured? The more annotators rely on *subjective* judgment, the more human bias seeps in. |
| **Feature engineering** | Does any feature carry sensitive info, or cause **disparate impact** — widely different outcomes for different groups from a seemingly neutral process? (A hiring model can discriminate by race via *correlated* proxies like zip code, even without using race.) |
| **Model's objective** | Does it optimize fairness for *all* users, or skew toward the majority group? |
| **Evaluation** | Are you doing **fine-grained, slice-based** evaluation across groups — and is your eval *data* itself fair and adequate? |

To fight disparate impact you can apply techniques like Feldman et al.'s **disparate-impact remover** (in IBM's **AI Fairness 360 / AIF360**) or surface hidden bias in correlated variables with the **Infogram** method (in H2O).

#### 2. Understand the data-driven approach's limits

ML is *data-driven*, but **data alone is not enough.** Data is about *people* — with socioeconomic and cultural context a dataset can't capture. Reliance on data creates **blind spots**, and seeing them means crossing disciplinary and organizational boundaries to account for the **lived experience** of those affected. A fair auto-grader, for instance, would require domain experts to explain how socioeconomic factors shaped the historical performance data in the first place.

#### 3. Understand the trade-offs between objectives

ML literature often pretends that improving one property (say, accuracy) holds all others fixed. In reality, **improving one property degrades another** — and the damage is rarely spread evenly.

::: callout math "Two trade-offs where the cost lands on the vulnerable"
- **Privacy vs. accuracy.** **Differential privacy** shares dataset-level patterns while hiding individuals — but the *more* privacy it provides, the *lower* the accuracy. Crucially, that accuracy drop is **not uniform**: Bagdasaryan & Shmatikov (2019) showed it falls hardest on **under-represented classes and subgroups.**
- **Compactness vs. fairness.** Model compression (Chapter 7) can cut parameters ~90% with "minimal" average accuracy loss. But Hooker et al. found that loss is **concentrated** in a narrow slice of the data — compression *amplifies harm* when the protected feature (sex, race, disability) sits in the long tail. And not all methods are equal: **pruning** causes far more disparate impact than the quantization techniques they tested.
:::

The takeaway: know these trade-offs so you can decide *consciously*. If your system is compressed or differentially private, **budget extra resources to audit** it for unintended harm.

#### 4. Act early — bias compounds

Picture a contractor who uses cheap cement and an owner who skips supervision to move fast. The building finishes on time — then cracks within a year and must be demolished, costing far more than was ever saved. ML ethics works the same way: bypass ethical issues early to save time, and the risks resurface later at much higher cost (exactly the Ofqual and Strava stories). A NASA study found the **cost of fixing an error rises by an order of magnitude at every stage** of the lifecycle. The earlier you think about who your system affects and what biases it carries, **the cheaper it is to fix.**

#### 5. Create model cards

**Model cards** are short documents that ship *with* a trained model, describing how it was trained and evaluated, the **context** it's meant for, and its **limitations.** Their goal (per Mitchell et al.'s "Model Cards for Model Reporting") is to **standardize ethical reporting** so stakeholders can compare candidate models not just on accuracy but on ethical, inclusive, and fair dimensions. A card typically reports:

- **Model details** — who built it, date, version, type, training algorithm, license, contact
- **Intended use** — primary uses, primary users, and explicit **out-of-scope** uses
- **Factors** — relevant demographic, phenotypic, environmental, or technical factors
- **Metrics** — performance measures, decision thresholds, variation approaches
- **Evaluation & training data** — datasets, motivation, preprocessing
- **Quantitative analyses** — unitary *and* intersectional results
- **Ethical considerations** and **caveats & recommendations**

::: callout note "Automate your model cards"
Cards are most valuable when the *users* of a model aren't its *developers.* But a card must be **updated every time the model changes** — crippling overhead if done by hand for a frequently-retrained model. So **auto-generate** them, via the model-card features in TensorFlow, Metaflow, or scikit-learn, or your own tooling. Because a card's contents overlap heavily with a **model store**, the author expects model stores to start generating cards automatically.
:::

#### 6. Establish process, tools, and ongoing learning

The more **ad hoc** your responsible-AI work, the more room for error — so put **systematic processes** in place. Build a portfolio of **internal tools** different stakeholders can reach; reference big-company tool sets like **Google's responsible-AI best practices** and IBM's open-source **AI Fairness 360**; and consider **third-party audits.** Finally, AI moves fast — new bias sources and new mitigation techniques appear constantly — so **stay current** by following groups like ACM FAccT, the Partnership on AI, the Alan Turing Institute's Fairness/Transparency/Privacy group, and the AI Now Institute.

## Summary

We've reached the end. ML solutions are deeply technical, but they cannot be confined to the technical domain — they are **built by humans, used by humans, and leave marks on society.** This last chapter deliberately turned from pipelines to people.

We saw how ML's **probabilistic, mostly-correct, high-latency** nature reshapes user experience: inconsistency frustrates ("I just saw that option — where did it go?"), unfixable predictions are useless (so show **several** candidates and let a human choose), and slow queries call for a **fast backup** that fails smoothly. We weighed two ways to staff a project — **siloed teams** (simpler hiring, but heavy hand-off friction) versus **end-to-end data scientists** (powerful, but only viable atop excellent tooling). And we closed on what the author calls the most important topic in the book: **responsible AI** — not an abstraction or a compliance checkbox, but an urgent, everyday practice. The framework here will help you meet compliance, but it is no substitute for the most important question of all: *should this product be built at all?*

::: callout key "The thread that ties the whole book together"
From Chapter 1's "the model is just one box" to this chapter's "the model is just one *actor*" — the through-line is the same. **Designing an ML system means designing for everything around the model**: the data that feeds it, the infrastructure that runs it, the stakeholders who shape it, the users who live with its predictions, and the society that absorbs its impact. Master that systems view and you'll still be designing great ML long after today's hottest algorithm is a footnote.
:::

## Key takeaways

::: takeaways
- ML's **probabilistic** nature means predictions can flicker; pin them with **consistency rules** when stability matters more than squeezing out accuracy (the **consistency–accuracy trade-off**).
- **"Mostly correct"** predictions only help if users can act on them — when they can't, **render multiple candidates** so a non-expert can pick a good one (**human-in-the-loop**).
- For queries the main model is too slow on, route to a **fast backup** (heuristic, simple model, or cache) — a **smooth fail** that exploits the **speed–accuracy trade-off** to get both.
- Involve **subject-matter experts across the whole lifecycle** — problem framing, features, evaluation, UI — not just labeling; empower them with **no-/low-code** tools.
- Prefer **end-to-end data scientists** over siloed hand-offs, but only once you've given them **infrastructure that abstracts away containers, distributed compute, and failover.**
- **Responsible AI** = fairness, privacy, transparency, accountability. The Ofqual grader failed on **objective, evaluation, and transparency**; Strava showed that **"anonymized" data can still leak** — so make **opt-in the default.**
- Build responsible AI deliberately: **find bias sources** at every stage, **respect data's limits**, **weigh trade-offs** (privacy↔accuracy, compactness↔fairness — costs land on the vulnerable), **act early** (bias compounds), and ship **model cards.**
:::
