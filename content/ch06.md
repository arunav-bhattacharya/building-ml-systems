By now you've created training data (Chapter 4) and engineered features from it (Chapter 5). This is the moment the work pays off: you finally get to feed those features into an **ML algorithm** and turn months of data wrangling into actual predictions. For many practitioners, this is the most fun part of the whole project — you get to play with algorithms, even the newest ones, and watch a system come to life.

But "the algorithm part" is bigger than picking a model and hitting *train*. This chapter walks through the full arc of **model development** — selecting an algorithm, combining models into ensembles, tracking and versioning experiments, scaling training across machines, and even automating model design with AutoML — and then the equally crucial discipline of **offline evaluation**: how to know, *before* you ship, whether a model is actually any good.

A note on scope, mirroring the book: this chapter assumes you already know how common algorithms work — linear models, decision trees, k-nearest neighbors, neural networks. It discusses the *techniques and decisions surrounding* those algorithms, not their internals. If the names feel unfamiliar, skim an intro-ML resource first.

::: callout note "Development is iterative"
Model development is a **loop**, not a line. After each iteration you compare against previous runs and ask: *is this version good enough for production?* The evaluation methods in the second half of this chapter are how you answer that question honestly — and they matter just as much as the modeling itself.

## Model Development and Training

This first half covers everything you need to develop and train a model: choosing among candidate algorithms, building ensembles, tracking and versioning experiments, scaling up with distributed training, and finally the advanced frontier of AutoML — using ML to choose the model for you.

### Evaluating ML Models

Any problem has many possible solutions. Should you start with the logistic regression you already know? Try the shiny new state-of-the-art model? Trust the senior colleague who swears gradient-boosted trees always work for this task? With infinite time and compute you'd try everything — but you don't have that, so you have to be **strategic about which models you evaluate.**

First, a myth to bust. Because nearly all of the last decade's headline AI progress came from ever-bigger neural networks, it's easy to assume deep learning has *replaced* classical ML. It hasn't. Many recommender systems still run on **collaborative filtering and matrix factorization**. Tree-based methods — especially **gradient-boosted trees** — still power countless classification tasks with tight latency budgets. And the two families work *together*: a k-means model might generate features for a neural net; a pretrained BERT or GPT might generate embeddings that feed a humble logistic regression.

You also don't choose from *every* model in existence — you narrow to the handful suited to your **problem type.** "Detect toxic tweets" is text classification → naive Bayes, logistic regression, RNNs, transformer models (BERT, GPT). "Detect fraudulent transactions" is anomaly detection → k-nearest neighbors, isolation forests, clustering, neural nets. **Knowing the common task → algorithm mappings is the core skill here.**

::: callout key "Evaluate more than accuracy"
Different algorithms need different amounts of **labeled data, compute, and training time**, have different **inference latency**, and differ wildly in **interpretability** (non-neural models are usually far easier to explain). A simple logistic regression may score lower than a complex net, yet still win because it needs less data, trains in minutes, deploys trivially, and can *tell you why* it predicted what it did.
:::

The book deliberately skips a head-to-head ranking of algorithms — any such comparison is obsolete the moment a new model drops. (In 2016, LSTM-based seq2seq ruled NLP; two years later transformers had largely displaced it.) Instead, here are six durable tips for deciding what to try next.

**1. Avoid the state-of-the-art trap.** People crave SOTA models — they *feel* like the best answer, they make a business look cutting-edge, and they're more fun to build. But "state of the art" usually just means *it beat other models on some static academic benchmark.* That says nothing about whether it's fast enough, cheap enough, or even better *on your data.* If a simpler, cheaper solution solves your problem, use it.

**2. Start with the simplest models.** *Simple is better than complex.* Simplicity buys you three things: simple models **deploy early** (validating that your prediction pipeline matches your training pipeline); they're **easier to understand and debug** as you layer on complexity; and the simplest model is a **baseline** for the fancier ones. One subtlety — *simplest* isn't always *least effort*. A pretrained BERT via Hugging Face is complex internally but trivial to start with. It's fine to use it, though you should still confirm it actually beats simpler options — and note that low-effort-to-start can mean high-effort-to-improve, whereas a simple model leaves you lots of room to climb.

**3. Avoid human bias in comparing models.** An engineer spends two weeks and reports BERT beats gradient-boosted trees by 5%, so you ship BERT. Months later a new hire revisits the trees and finds they now win. What happened? An engineer excited about an architecture **runs far more experiments** on it, tuning features and hyperparameters until it shines. The fix: **compare architectures under comparable setups.** If you ran 100 experiments for one, run 100 for the other — anything less isn't a fair fight.

**4. Evaluate good-now vs good-later.** The best model *today* may not be the best in two months. A tree model might win now because data is scarce; double your data and a neural net could pull ahead. A **learning curve** — performance plotted against number of training samples — won't predict the exact gain, but it tells you whether *more data is likely to help at all.* The book's example: offline, a collaborative-filtering model beat a simple neural net for recommendations — but the net could update on each incoming example while CF had to re-factor its whole matrix. Deployed together, the continually-trained net overtook CF within two weeks.

**5. Evaluate trade-offs.** Choosing a model means accepting trade-offs; knowing which matters most guides the choice.

| Trade-off | Example | Lean toward… |
|---|---|---|
| **False positives vs false negatives** | Fingerprint unlock — a wrong "authorized" is dangerous | Fewer **false positives** |
| | COVID-19 screening — a missed case is dangerous | Fewer **false negatives** |
| **Compute vs accuracy** | A complex net needs a GPU to hit acceptable latency | Cheaper compute *or* higher accuracy |
| **Interpretability vs performance** | A complex model scores higher but can't explain itself | Whichever the stakeholders require |

**6. Understand your model's assumptions.** *"All models are wrong, but some are useful"* (George Box, 1976). The real world is too complex to capture exactly, so every model leans on **assumptions** — and your model is only useful if your data satisfies them.

::: callout math "Common model assumptions"
- **Prediction:** it's possible to predict output $Y$ from input $X$ at all.
- **IID:** examples are independent and identically distributed (assumed by neural nets).
- **Smoothness:** similar inputs map to similar outputs — close $X$ → proportionally close $Y$ (assumed by all supervised learning).
- **Tractability:** the latent representation $P(Z \mid X)$ is computable (assumed by generative models).
- **Boundaries:** decision boundaries are linear (linear classifiers).
- **Conditional independence:** features are independent given the class (naive Bayes).
- **Normality:** the data is normally distributed (many statistical methods).
:::

### Ensembles

Once you have a single solid model, one technique reliably squeezes out more performance: combine several models into an **ensemble**, where each member is a **base learner.** For spam detection you might run three different models and take the **majority vote** — at least two say SPAM, it's SPAM.

Ensembles dominate competitive ML: 20 of 22 winning Kaggle solutions in 2021 used them, as did the top 20 solutions on the SQuAD 2.0 leaderboard. They're **less common in production** — more complex to deploy, harder to maintain — but still worth it where a tiny accuracy gain means big money, like ad click-through prediction.

::: callout example "Why ensembling works — the intuition"
Take three *uncorrelated* spam classifiers, each 70% accurate. The majority vote is correct whenever at least two are right:

| Outcome | Probability | Ensemble |
|---|---|---|
| All three correct | 0.7³ = **0.343** | Correct |
| Exactly two correct | 3 × (0.7 × 0.7 × 0.3) = **0.441** | Correct |
| Exactly one correct | 3 × (0.3 × 0.3 × 0.7) = 0.189 | Wrong |
| None correct | 0.3³ = 0.027 | Wrong |

Accuracy = 0.343 + 0.441 = **78.4%** — an 8-point jump from 70%.
:::

The catch: this math **only holds if the base learners are uncorrelated.** If all three always agree, the ensemble is no better than one model. So the rule is **the less correlated the base learners, the better the ensemble** — which is why ensembles deliberately mix very different model types (e.g., a transformer + an RNN + a gradient-boosted tree). The three ways to build one are **bagging, boosting, and stacking.**

#### Bagging

**Bagging** (bootstrap **agg**regat**ing**) improves both training stability and accuracy, mainly by **reducing variance and curbing overfitting.** Instead of training one model on the whole dataset, you sample *with replacement* to create multiple datasets called **bootstraps**, and train one model on each. (Sampling with replacement makes each bootstrap independent.) For classification, the final answer is the **majority vote**; for regression, the **average.** Bagging helps **unstable** methods — neural nets, decision/regression trees, linear-regression subset selection — but can mildly *hurt* stable ones like k-NN. A **random forest** is the classic example: a collection of decision trees built with both bagging *and* feature randomness (each tree may only use a random subset of features).

#### Boosting

**Boosting** is a family of *iterative* algorithms that convert **weak learners into a strong one.** Every learner trains on the same samples, but the samples are **reweighted between rounds** so each new learner focuses on what its predecessors got wrong:

1. Train the first weak classifier on the original data.
2. **Reweight** samples by how well it did — misclassified ones get higher weight.
3. Train the second classifier on the reweighted data; add it to the ensemble.
4. Reweight again based on the *ensemble's* errors.
5. Train the third; add it. Repeat as needed.
6. The final strong classifier is a **weighted combination** — lower-error classifiers get higher weight.

The canonical example is the **gradient boosting machine (GBM)**, which builds the model stage-by-stage from weak decision trees and generalizes boosting to any differentiable loss. **XGBoost** — a GBM variant — was for years the go-to for competition winners, used in everything from ranking to the discovery of the Higgs Boson. Many teams now reach for **LightGBM**, a distributed gradient-boosting framework whose parallel learning trains faster on large datasets.

#### Stacking

**Stacking** trains base learners on the training data, then trains a **meta-learner** that combines their outputs into the final prediction. The meta-learner can be a simple heuristic (majority/average vote) or another model entirely — a logistic or linear regression that *learns* how to weight the base learners. Because it learns the combination rule rather than hard-coding it, stacking can exploit each base learner's strengths.

| Method | How models relate | Mainly fixes | Example |
|---|---|---|---|
| **Bagging** | Parallel, independent bootstraps | Variance | Random forest |
| **Boosting** | Sequential, reweighted data | Bias | XGBoost, LightGBM |
| **Stacking** | Parallel, combined by a meta-learner | Both | Kaggle-winning stacks |

### Experiment Tracking and Versioning

During development you'll try **many** architectures and models. Two can differ by a single hyperparameter — a learning rate of 0.003 vs 0.002 — yet perform *dramatically* differently. So you must record everything needed to recreate a run and its **artifacts** (any file produced during an experiment — loss curves, eval graphs, logs, intermediate outputs). This lets you compare experiments, pick the best, and *understand* how small changes move performance.

Two related practices:

- **Experiment tracking** — recording the progress and results of a run.
- **Versioning** — logging every detail of a run so you can recreate or compare it later.

They go hand in hand, and tools have converged: tracking tools like **MLflow** and **Weights & Biases** added versioning; versioning tools like **DVC** added tracking.

::: callout note "Babysitting the training process"
A big part of training is **watching it happen.** Loss might not decrease; the model might overfit or underfit; weights might fluctuate; neurons might die; you might run out of memory. Tracking exists to catch these *and* to confirm the model is learning anything useful at all.
:::

**What to track.** Early on you might track only loss and speed; modern dashboards track far more. Worth considering per experiment:

- **Loss curves** for the train split and each eval split.
- **Performance metrics** you care about on all non-test splits (accuracy, F1, perplexity).
- A **log of samples, predictions, and ground-truth labels** — invaluable for ad-hoc analysis and sanity checks.
- **Speed** — steps per second, or tokens per second for text.
- **System metrics** — memory usage, CPU/GPU utilization — to spot bottlenecks and wasted resources.
- **Parameter/hyperparameter values over time** — e.g., learning rate (if scheduled), gradient norms (if clipping), weight norms (if using weight decay).

In theory, track everything — most of it you'll never look at, but when something breaks, one of those signals may be the clue you need (tracking gives you **observability**). In practice, today's tooling makes *too much* tracking overwhelming, and noise can distract you from the signals that matter.

**Versioning data, not just code.** Because ML is *part code, part data*, you must version both. Code versioning is industry standard; **data versioning is like flossing — everyone agrees it's good, few do it.** It's genuinely hard:

- Data is far larger than code, so code's strategies don't transfer. A code **diff** compares line by line — meaningful because lines are short. A "line" of data (especially binary) can be a million characters long, so "this 1M-char line differs from that one" tells you nothing.
- You can't always **duplicate** a dataset to keep old versions, or fit it on each developer's laptop.
- It's unclear what a **diff** even *is* — any file change? Only additions/removals? A changed checksum? (As of 2021, DVC registers a diff only when the directory checksum changes or a file is added/removed.)
- **Merge conflicts** don't make sense: if dev 1 trains model A on data version X and dev 2 trains model B on version Y, merging X and Y into Z is meaningless — no model corresponds to Z.
- **Regulations** like GDPR can make it *illegal* to recover old data versions (you may be required to delete user data on request).

::: callout warning "Tracking helps reproducibility — but doesn't guarantee it"
Even with aggressive tracking and versioning, frameworks and hardware can inject **nondeterminism** (e.g., CUDA atomic operations reorder floating-point ops between runs), making exact replication impossible without knowing *everything* about the environment. We run so many experiments precisely because we still treat ML as a black box — we can't yet *reason* about which config will win, so we brute-force it.
:::

### Distributed Training

As models balloon in size, training at scale becomes unavoidable. Often the data **won't fit in memory** — common with CT scans, genome sequences, or large-language-model text corpora. When that happens, preprocessing, shuffling, and batching must run **out-of-core** and in parallel. When a single *sample* is huge, you may be stuck with a tiny batch size, which **destabilizes gradient descent.** And when a sample won't fit at all, techniques like **gradient checkpointing** trade compute for memory — the authors of one such package fit models **10× larger** on the same GPU for only a **20% slowdown.**

#### Data parallelism

The most common approach, supported by every modern framework: **split the data across machines, train a copy of the model on each, and accumulate the gradients.** Two issues arise.

**How do you combine gradients?** If you wait for *all* machines to finish each step — **synchronous SGD** — then **stragglers** stall the whole system. Worse, the more machines, the more likely *some* worker runs slow in a given step. (Good news: many algorithms now mitigate this.) Alternatively, update weights from each machine's gradient *separately* — **asynchronous SGD** — but then **gradient staleness** bites: one machine's gradient may change the weights *before* another's arrives, so the second is computed against outdated weights.

In theory async SGD needs more steps to converge. In practice, when there are many weights, gradient updates are **sparse** — each modifies only a small slice of parameters — so two machines rarely touch the same weights, staleness fades, and sync and async converge similarly.

**Batch size explodes.** Spread the model across many machines and the effective batch size balloons — 1,000 machines at batch 1,000 each = 1M (GPT-3 175B used a batch of 3.2M). Bigger batches mean fewer steps, so the tempting fix is to **scale up the learning rate** — but too large a rate destabilizes convergence, and past a point bigger batches give **diminishing returns.** A final wrinkle: the main worker often uses more resources than the others, so you must **balance the workload** (crudely, a smaller batch on the main worker).

#### Model parallelism

When the model itself is too big for one machine, **model parallelism** puts **different components on different machines** — machine 0 runs layers 1–2, machine 1 runs layers 3–4, or some machines do the forward pass while others do the backward pass.

The name misleads: the parts don't always run *in parallel.* Split a giant matrix across two machines and the halves *can* compute simultaneously. But put layer 1 on machine 1 and layer 2 on machine 2 — and layer 2 needs layer 1's output — so **machine 2 must wait** for machine 1.

::: callout key "Pipeline parallelism keeps machines busy"
**Pipeline parallelism** restores the parallelism by breaking each machine's work into chunks. Split a mini-batch into **micro-batches**: machine 1 computes layer 1 on micro-batch 1, hands it off, then immediately starts micro-batch 2 — while machine 2 works on micro-batch 1. Now the machines overlap instead of idling in sequence.
:::

Data and model parallelism aren't mutually exclusive — many teams combine both for better hardware utilization, though wiring them together takes serious engineering effort.

### AutoML

There's a joke that a great ML researcher automates *themselves* out of a job. It stopped being funny at the 2018 TensorFlow Dev Summit, when Jeff Dean unveiled **AutoML** — Google's bet to replace ML expertise with **100× more compute.** Why pay 100 researchers to hand-tune a suboptimal model when you could spend that money searching for the optimal one? AutoML is **automating the process of finding ML algorithms** for real-world problems. It runs on a spectrum from mild to wild.

#### Soft AutoML: hyperparameter tuning

The mildest and most popular form. A **hyperparameter** is a user-supplied value that controls learning — learning rate, batch size, number of hidden layers/units, dropout probability, Adam's $\beta_1$ and $\beta_2$, even the bit-width of quantization. The *same* model with different hyperparameters can perform *drastically* differently on the *same* data — one 2018 paper showed that **well-tuned weaker models beat fancier ones.** The goal: find the optimal hyperparameter set within a **search space**, scoring each on a **validation set.**

Despite knowing this matters, many still tune by gut — the dominant method being **"graduate student descent,"** a grad student fiddling until it works. Better: use systematic tools — `auto-sklearn`, Keras Tuner, Ray Tune — with **random search, grid search, or Bayesian optimization.** Tune your *sensitive* hyperparameters most carefully, since the model reacts more to some than others.

::: callout warning "Never tune on your test split"
Pick hyperparameters by **validation-split** performance, then report final numbers on the **test split.** Tuning on the test split overfits your model to it and inflates your reported performance — you'll be lying to yourself.
:::

#### Hard AutoML: architecture search and learned optimizers

Take it further: treat *components of the model* — or the *whole model* — as hyperparameters. The size of a convolution, whether to add a skip connection: all searchable. You hand the algorithm the **building blocks** (convolutions, linear, activations, pooling, identity, zero) and let it assemble them. This is **neural architecture search (NAS).** A NAS setup has three parts:

- **Search space** — the possible architectures: which building blocks, and the rules for combining them.
- **Performance estimation strategy** — a cheap way to judge a candidate *without* training it to convergence (training 1,000 candidates fully would be ruinously expensive).
- **Search strategy** — how to explore. Random search is too costly even here; common choices are **reinforcement learning** (reward choices that improve the estimate) and **evolution** (mutate an architecture, keep the best, mutate again).

The same idea reaches the **optimizer.** Normally a learning procedure (gradient descent) uses a hand-designed optimizer — Adam, Momentum, SGD — to turn gradients into weight updates. You *could* search over optimizers in NAS, but they're sensitive and don't transfer across architectures. The exciting alternative: replace the update rule with a **neural network** — a **learned optimizer.** Train it once across thousands of tasks and reuse it everywhere; Metz et al.'s learned optimizer generalized to new datasets, domains, *and* architectures. The beautiful twist — a learned optimizer can train a *better* learned optimizer, an algorithm that improves itself.

::: callout note "Why care about AutoML even if you can't afford it?"
The up-front cost of NAS or learned optimizers is so high that only a handful of companies worldwide can run it. But it matters to everyone because (1) the **resulting models work off-the-shelf** on many real tasks, saving production time and cost — Google's **EfficientNets** beat SOTA accuracy with up to **10× better efficiency** — and (2) they may solve tasks previously impossible with hand-designed architectures.
:::

## Model Offline Evaluation

A question the author hears constantly: *"How do I know our ML models are any good?"* One company deployed intrusion detection to 100 drones but had **no way to measure missed intrusions** — and so couldn't tell if one algorithm beat another. Lacking a way to evaluate your system won't necessarily *kill* your project, but it makes finding the best solution impossible and makes it harder to convince leadership to adopt ML. Partner with the business team to define metrics that map to *business* value.

Ideally you'd evaluate the same way in development and production — but usually you can't, because **production has no ground-truth labels.** Sometimes you can *infer* labels from user feedback (did they click the recommendation?), though with many biases; otherwise you lean on monitoring (Chapter 8). This section covers what you *can* do before deployment: pick **baselines**, then run **evaluation methods** that go beyond a single accuracy number.

### Baselines

Someone once told the author her generative model hit an **FID score of 10.3** on ImageNet. Was that good? Impossible to say in isolation. Another engineer proudly reported an **F1 of 0.90** — until asked, "compared to random?" His positive class appeared 90% of the time, so a model that *just guessed positive* would also score ~0.90. His model might be **no better than chance.**

::: callout key "Metrics are meaningless without a baseline"
A raw evaluation number tells you almost nothing. You must know **what you're comparing against.** Five baselines recur across use cases.
:::

Conversely, a next-word predictor far worse than a native speaker is *bad*, yet *useful* if it helps you type faster sometimes. Your model doesn't have to beat the existing solution to be worth shipping — a slightly worse model can win if it's much cheaper or easier to run.
:::

### Evaluation Methods

Academia fixates on performance metrics. Production needs more: models that are **robust, fair, calibrated, and sensible.** Here are the methods that probe those qualities.

#### Perturbation tests

Students built a COVID-cough classifier that aced training data (clean two-second clips from hospitals) but predicted near-randomly on real users — whose recordings carried background music, chatter, varied mic quality, and inconsistent timing. **Real inputs are noisier than development inputs**, and the model best on clean data often *isn't* best on noisy data.

The fix: **perturb your test set** — add background noise, randomly clip the audio — to simulate real-world variance, and prefer the model that holds up on the **perturbed** data. The more sensitive your model is to noise, the harder it is to maintain (a user switching phones could degrade it) and the more vulnerable it is to **adversarial attacks.**

#### Invariance tests

A Berkeley study found 1.3 million creditworthy Black and Latino applicants were rejected for mortgages because of their race — when researchers kept income and credit scores but **deleted the race features**, the applications were *accepted.* **Some input changes must *not* change the output:** race shouldn't affect a loan, a name shouldn't affect résumé screening, gender shouldn't affect pay. To catch such bias, hold inputs fixed and **flip the sensitive attribute** to see if the output moves. Better still — **exclude sensitive features from training entirely** (and it may be legally required).

#### Directional expectation tests

The mirror image: some input changes **should** move the output in a **predictable direction.** For house-price prediction, increasing lot size shouldn't *lower* the price, and shrinking square footage shouldn't *raise* it. If the output moves the *wrong* way, your model may be learning the wrong thing — investigate before deploying.

#### Model calibration

A subtle but crucial idea. If a model says an event has **70% probability**, then across all the times it says "70%," the event should actually happen **70% of the time.** If it predicts team A beats team B with 70% probability but A only wins 60% of such matchups, the model is **miscalibrated** — it should be saying 60%.

::: callout example "Why calibration matters"
**Recommendations:** user A watches romance 80% of the time and comedy 20%. An *uncalibrated* "show the single most likely movie" recommender serves *only* romance. A **calibrated** one mirrors actual habits — 80% romance, 20% comedy.

**Click prediction:** to merely *rank* ad A (10%) above ad B (8%), calibration is unnecessary. But to **estimate total clicks**, it's essential — if the model says 10% but reality is 5%, your click forecast is wildly off. Between two equally-ranking models, prefer the **better-calibrated** one.
:::

To measure it, **count**: for each predicted probability $X$, find the actual frequency $Y$ of it coming true, and plot $X$ vs $Y$ — a perfectly calibrated model lands on the line $X = Y$. (`sklearn.calibration.calibration_curve` does this.) To *fix* miscalibration, **Platt scaling** is the common method (`sklearn.calibration.CalibratedClassifierCV`).

#### Confidence measurement

While most metrics describe **average** performance, confidence measurement is **per-sample** — a usefulness threshold for each individual prediction. Showing every prediction, even ones the model is unsure about, ranges from **annoying** (a smartwatch insisting you're running when you're walking fast) to **catastrophic** (predictive policing flagging an innocent person). So: how do you measure certainty per prediction? What threshold must a prediction clear to be shown? And below it — discard, defer to a human, or ask the user for more info? System-level metrics gauge overall health, but **sample-level** confidence is what you need when every single prediction must count.

#### Slice-based evaluation

**Slicing** splits your data into subgroups and measures performance **on each separately.** A frequent mistake: obsessing over coarse aggregate metrics (overall F1, overall accuracy) and ignoring slices. Two problems follow.

**Problem 1 — the model does worse on a slice where it shouldn't.** Suppose the majority subgroup is 90% of the data:

| | Majority acc. | Minority acc. | Overall acc. |
|---|---|---|---|
| **Model A** | 98% | 80% | **96.2%** |
| **Model B** | 95% | 95% | **95.0%** |

Looking only at overall accuracy, you'd ship **Model A** — until users discover it's **biased against the minority subgroup** (which may map to an underrepresented demographic). Aggregate focus is harmful twice over: it risks public backlash *and* blinds you to improvement — seeing the slices, you might lift Model A's minority performance (raising overall too), or at least make a better-informed deployment choice.

**Problem 2 — the model does equally on slices where it *should* differ.** Some slices matter more. In churn prediction, **paid users** are more critical than free ones. Optimizing overall performance can quietly hurt the critical slice.

::: callout warning "Simpson's paradox: aggregates can outright lie"
**Simpson's paradox** is a trend that appears in subgroups but **reverses when combined.** Model B can beat Model A overall while Model A beats B on *every* subgroup:

| | Group A | Group B | **Overall** |
|---|---|---|---|
| **Model A** | **93%** (81/87) | **73%** (192/263) | 78% (273/350) |
| **Model B** | 87% (234/270) | 69% (55/80) | **83%** (289/350) |

(Real numbers from a 1986 kidney-stone study.) The same reversal hit **Berkeley's 1973 admissions**: men were admitted at a higher *overall* rate, suggesting bias against women — yet women had *higher* admission rates in **four of six departments.** Aggregation hid the truth. **Look at slices.**
:::

Slice-based evaluation improves performance overall *and* on critical data, surfaces bias, and can even reveal **non-ML bugs** — the author's team once found a model "great overall, terrible on mobile" because a button was half-hidden on small screens. Even when you doubt slices matter, fine-grained understanding builds the confidence to convince stakeholders to trust your model.

How do you find critical slices? Slicing is still **more art than science**, via three approaches:

| Approach | How it works |
|---|---|
| **Heuristics-based** | Slice by domain knowledge — mobile vs desktop, browser, geography |
| **Error analysis** | Manually inspect misclassified examples and spot patterns (how the mobile bug was found) |
| **Slice finder** | Algorithms (beam search, clustering, decision trees) generate slice candidates, prune the bad ones, and rank the rest |

Once you know your critical slices, you'll need **enough correctly-labeled data for each** — your evaluation is only as good as your evaluation data.

## Summary

This chapter covered the **ML algorithm** part of an ML system — for many, the most fun stretch, where data and feature engineering finally turn into predictions you can test. We started with how to **select models**: rather than ranking every architecture (a fool's errand as new ones appear), we laid out the six tips and the trade-offs to weigh against your objectives and constraints. We then covered **ensembles** — bagging, boosting, stacking — widely used in competitions, less so in production.

Because development means running *many* experiments, we covered **tracking and versioning** (often skipped because it feels like a chore — which is exactly why tooling to automate it matters). As models grow, **distributed training** — data, model, and pipeline parallelism — becomes essential and demands real systems engineering. We pushed into **AutoML**, from hyperparameter tuning to architecture search and self-improving learned optimizers.

Finally, **offline evaluation**: metrics mean nothing without a **baseline**, and overall accuracy hides critical failures — so we covered perturbation, invariance, directional, calibration, confidence, and slice-based methods to sanity-check a model before it ships. But no matter how thorough your offline evaluation, you can't be *certain* of a model's behavior until it's deployed — which is exactly where the next chapter goes.

## Key takeaways

::: takeaways
- **Don't chase state of the art.** Start with the simplest model that could work; classical ML (gradient-boosted trees, collaborative filtering) still dominates tabular and latency-sensitive problems. Weigh accuracy *and* data, compute, latency, and interpretability.
- When comparing architectures, **give each equal effort** — human enthusiasm, not the model, often decides "winners." And consider **good-later**, not just good-now (use learning curves).
- **Ensembles** boost performance, and the gain comes from **uncorrelated** base learners — bagging cuts variance (random forests), boosting cuts bias (XGBoost), stacking learns a meta-combiner.
- **Track and version code *and* data.** Tracking aids reproducibility but can't guarantee it (hardware nondeterminism). Data versioning is hard — diffs, size, merges, and GDPR all fight you.
- **Distributed training:** data parallelism splits data (sync SGD has stragglers; async SGD has stale gradients; huge batches give diminishing returns); model parallelism splits the model, and **pipeline parallelism** with micro-batches keeps machines busy.
- **AutoML** runs from soft (hyperparameter tuning — *never* tune on the test split) to hard (NAS and learned optimizers — costly, but the resulting models work off-the-shelf with great efficiency).
- A metric is meaningless without a **baseline** (random, heuristic, zero rule, human, existing). A *good* system isn't always *useful*.
- Go beyond aggregate accuracy: **perturbation** (robustness), **invariance** (fairness), **directional** (sanity), **calibration** (honest probabilities), **confidence** (per-sample), and **slice-based** evaluation — because **Simpson's paradox** means aggregates can hide, or even reverse, subgroup truth.
:::

## Test your knowledge

```flashcards
Q: What is the "state-of-the-art trap," and why should you avoid it?
A: Assuming the newest, highest-benchmark model is best for *your* problem. SOTA only means it beat others on some static academic dataset — it says nothing about whether it's fast enough, cheap enough, or even more accurate on your data. Start with the simplest solution that works.
---
Q: Why do ensembles improve performance, and what condition must hold?
A: Combining models (e.g., majority vote) lowers the chance they're all wrong at once. Three uncorrelated 70%-accurate classifiers vote to ~78.4% accuracy. But the gain **only holds if base learners are uncorrelated** — if they always agree, the ensemble is no better than one model.
---
Q: Bagging vs boosting vs stacking — one line each.
A: **Bagging** trains models in parallel on bootstrapped samples and votes (reduces variance; e.g., random forest). **Boosting** trains weak learners sequentially on reweighted data, focusing on prior errors (reduces bias; e.g., XGBoost). **Stacking** trains a meta-learner to combine base learners' outputs.
---
Q: In data parallelism, contrast synchronous and asynchronous SGD.
A: **Sync SGD** waits for all machines before updating — correct but **stragglers** stall everything. **Async SGD** updates from each machine separately — faster but suffers **gradient staleness** (weights change before a machine's gradient arrives). With many weights, updates are sparse, so the two converge similarly in practice.
---
Q: What does it mean for a model to be "calibrated," and when does it matter?
A: When it predicts probability X, the event actually happens X% of the time. It's needed when you use the probabilities themselves — estimating total ad clicks, or making recommendations representative of real habits (80% romance / 20% comedy) — but not if you only need a ranking.
---
Q: What is Simpson's paradox, and why does it make slice-based evaluation essential?
A: A trend that holds within every subgroup but **reverses when groups are combined** — Model A can beat Model B on each slice yet lose overall (or vice versa). Aggregate metrics can hide or contradict reality, so you must evaluate per slice to choose models and detect bias.
---
Q: Name the five baselines you might compare a model against.
A: Random baseline, simple heuristic, zero rule (always predict the most common class), human baseline, and the existing solution being replaced.
---
Q: Why is data versioning harder than code versioning?
A: Data is far bigger than code (can't duplicate or fit locally), a line-by-line "diff" is meaningless for long/binary data, it's unclear what even counts as a diff, merges don't make sense (no model maps to a merged dataset), and GDPR may legally forbid recovering old versions.
```

- ( ) Pick whichever the senior engineer prefers
- ( ) Evaluate only on the training set
> Human bias creeps in when an engineer, excited about one architecture, runs far more experiments on it. A fair comparison runs comparable effort on each — if 100 experiments for one, then 100 for the other.

A classifier reports an F1 of 0.90 on a task where the positive class appears 90% of the time. What's the problem?
- ( ) F1 of 0.90 is always excellent
- (x) A model that always predicts positive would also score ~0.90, so it may be no better than random
- ( ) F1 cannot be computed on imbalanced data
- ( ) The model is overfitting the test split
> Metrics are meaningless without a baseline. Given the 90% positive rate, the random/zero-rule baseline already reaches ~0.90 — so the model must be checked against that before claiming success.

Pipeline parallelism is used to address which problem in model parallelism?
- ( ) Gradient staleness from asynchronous updates
- ( ) Batch sizes growing too large
- (x) Machines sitting idle because a later layer must wait for an earlier layer's output
- ( ) Data not fitting into memory
> When layer 2 (machine 2) needs layer 1's (machine 1) output, machine 2 idles. Pipeline parallelism splits each mini-batch into micro-batches so machines work on different micro-batches simultaneously instead of waiting.

- ( ) Model A is simpler to deploy

- ( ) Subgroup performance is irrelevant offline
> Aggregate metrics hide subgroup failures. Slice-based evaluation exposes Model A's weakness on the minority subgroup — a fairness risk and a missed improvement opportunity that overall accuracy conceals.

Which is the correct, safe way to use a test split when tuning hyperparameters?
- ( ) Tune directly on the test split to maximize the reported score
- (x) Tune on the validation split, then report final performance once on the test split

- ( ) Skip the validation split entirely
