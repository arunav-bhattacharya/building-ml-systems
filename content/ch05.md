In 2014, a Facebook paper with a wonderfully unglamorous title — *"Practical Lessons from Predicting Clicks on Ads"* — made a claim that has aged extremely well: **having the right features is the single most important thing** in building their ML models. Teams everywhere keep rediscovering the same lesson. Once you have a model that works at all, *better features* usually buy you a bigger jump in performance than any amount of clever hyperparameter tuning. A state-of-the-art architecture fed mediocre features will lose to a humble model fed great ones.

That is why a huge slice of every ML engineering and data science job is, quietly, **feature engineering**: deciding what information to feed the model and how to shape it. This chapter walks through the common operations, then spends real time on a subtle, project-killing bug that hides inside feature work — **data leakage** — and finishes with how to tell a *good* feature from a useless or dangerous one.

When people hear "feature engineering" they sometimes think of **feature stores**. Those are infrastructure for sharing and serving features across many ML applications, so they belong with the data-systems material later in the book. This chapter is about the *craft* of making features, not the platform that stores them.
:::

## Learned Features Versus Engineered Features

Students always ask the obvious question: *doesn't deep learning mean we no longer have to engineer features?* The promise is real — deep learning is sometimes literally called **feature learning**, because the network discovers useful representations on its own. But two caveats keep feature engineering alive and well:

1. We are nowhere near automating *all* feature extraction.
2. As of this book's writing, **most ML in production isn't deep learning at all** — it's still classical models that need hand-built features.

The classic illustration is a **comment-spam classifier**. Before deep learning, turning a comment into something a model could read meant a pipeline of hand-applied **natural language processing (NLP)** steps: lemmatize words, expand contractions, strip punctuation, lowercase everything, then chop the text into **n-grams** — contiguous runs of *n* items (here, words).

::: callout example "n-grams of \"I like food\""
The 1-grams are `["I", "like", "food"]`; the 2-grams are `["I like", "like food"]`. Pick n ∈ {1, 2} and the feature set becomes `["I", "like", "food", "I like", "like food"]`. Build a **vocabulary** mapping each n-gram to an index, and every comment becomes a fixed-length count vector. With the 7-entry vocabulary `[I, like, good, food, I like, good food, like food]`, the post *"I like good food"* encodes as `[1, 1, 1, 1, 1, 0, 1]`. That vector is what the model actually sees.
:::

This is brittle, iterative work — forget one step, or apply one that backfires, and you restart. Deep learning eased the pain for *text* and *images*: you can now tokenize raw text into words, build a vocabulary, feed one-hot vectors, and let the model learn the rest. Images go in essentially raw. **The feature engineering for those modalities got largely automated.**

But a real system needs more than the comment's text. To catch spam you'd also want signals about **the comment** (up/down-votes), **the user** (account age, posting frequency, their vote history), and **the thread** (view count — popular threads attract more spam). None of that is automatically extracted; *you* decide what to use and how to compute it. For something like TikTok's "what to watch next," feature counts climb into the **millions**, and domain-heavy tasks like fraud detection need genuine subject-matter expertise to invent good features. This is where much of the real work lives.

## Common Feature Engineering Operations

Because feature work is everywhere, the field has accumulated a toolbox of standard operations. This section covers the most useful ones: handling missing values, scaling, discretization, encoding categories, the old-school-but-mighty **feature crossing**, and the newer **positional embeddings**. It is not exhaustive — it's a strong starting kit.

### Handling Missing Values

The first thing you notice with production data is that values are *missing*. The crucial, often-overlooked insight: **not all missing values mean the same thing.** Consider predicting whether someone will buy a house in the next 12 months, from a table of age, gender, income, marital status, number of children, and job. There are three statistical types of "missing," and the textbook names are confusing, so anchor each to an example.

| Type | Full name | Why the value is missing | House-buying example |
|---|---|---|---|
| **MNAR** | Missing **N**ot **A**t **R**andom | The reason is the *true value itself* | High earners decline to report income — so a *blank income* hints at a *high income* |
| **MAR** | Missing **A**t **R**andom | The reason is *another observed variable*, not the value itself | Age is blank mostly for one gender, because that group dislikes disclosing age |
| **MCAR** | Missing **C**ompletely **A**t **R**andom | No pattern at all — pure chance | People occasionally forget to fill in "job," for no reason. **Rare in practice** — usually there *is* a hidden reason, so investigate |

You handle missing values two ways: **deletion** (drop them) or **imputation** (fill them).

**Deletion.** Interview candidates love deletion — not because it's better, but because it's *easy*. Two forms:

- **Column deletion** — drop a variable that is mostly blank. If "marital status" is >50% missing you may be tempted to ax it. The danger: you may throw away signal. Marital status correlates strongly with home-buying (married couples buy more), so deleting it can hurt accuracy.
- **Row deletion** — drop any sample with a missing value. This is safe only when the data is **MCAR** *and* the affected rows are a tiny fraction (say <0.1%). Never delete rows if it would erase 10% of your data.

Row deletion also has two subtler traps. With **MNAR** data, the blank *is* information — dropping the income-blank rows discards the very signal (likely-high earners) you wanted. With **MAR** data, deletion injects **bias**: drop every age-blank row and you may delete *every* member of one gender, so the model can no longer serve that group.

**Imputation.** If you'd rather not delete, you fill the blanks — and choosing the fill value is the hard part. Common practices:

- Fill with a **default** (missing job → empty string `""`).
- Fill with the **mean, median, or mode** (a July temperature reading is missing → use July's median temperature).

::: callout warning "Don't impute with a value that's already a real value"
A frontend stopped asking for age, so age arrived blank and the pipeline filled it with **0** — a value the model never saw in training, so it spat out garbage. And filling a missing *number of children* with **0** is just as bad: 0 is a legitimate count, so you can no longer tell "has no children" apart from "we don't know." Prefer a sentinel that can't be confused with a genuine reading.
:::

The honest bottom line: **there is no perfect way to handle missing values.** Delete and you risk losing information or amplifying bias; impute and you risk injecting bias, adding noise, or — worst of all — causing **data leakage** (covered soon).

### Scaling

In the house-buying data, **age** runs 20–40 while **annual income** runs 10,000–150,000. A model doesn't know these are different units — it just sees numbers, and because 150,000 dwarfs 40, it may hand income outsized importance regardless of which variable actually matters. **Feature scaling** fixes this by bringing features into similar ranges. It's one of the simplest wins available, and skipping it can produce gibberish — especially with classical algorithms like gradient-boosted trees and logistic regression.

::: callout math "Three ways to scale"
**Min-max to [0, 1]** — rescale by the observed range:
$$x' = \frac{x - \min(x)}{\max(x) - \min(x)}$$
The max maps to 1, the min to 0. For an arbitrary range $[a, b]$ (the author finds $[-1, 1]$ often beats $[0, 1]$):
$$x' = a + \frac{(x - \min(x))(b - a)}{\max(x) - \min(x)}$$

**Standardization** — if the variable looks roughly **normal**, give it zero mean and unit variance:
$$x' = \frac{x - \bar{x}}{\sigma}$$

**Log transform** — models struggle with **skewed** distributions, so applying $\log$ pulls in a long right tail and often boosts performance. It isn't a cure-all, and be cautious about analyzing log-transformed data as if it were the original.
:::

::: callout warning "Two things to remember about scaling"
1. **Scaling is a notorious source of data leakage** — the min/max/mean must be computed on the **training data only**, then reused at inference (more in *Data Leakage*).
2. Because it relies on **global statistics**, scaling drifts: if production data shifts away from the training distribution, the saved statistics go stale. That's a reason to **retrain often**.
:::

### Discretization (Binning)

**Discretization** — also called **quantization** or **binning** — turns a *continuous* feature into a *discrete* one by bucketing values. Suppose the model trained on incomes of 150,000, 100,000, 50,000, then meets `9,000.50` at inference. We *know* \$9,000.50 ≈ \$10,000, but the model only sees two different numbers. Bucketing income into **lower (<\$35K)**, **middle (\$35K–\$100K)**, and **upper (>\$100K)** lets the model learn three categories instead of infinitely many — easier, and most helpful when training data is **limited**. It works on already-discrete features too (e.g., grouping age into `<18, 18–22, 22–30, 30–40, 40–65, 65+`).

::: callout warning "Binning introduces hard edges"
Bucketing creates **discontinuities at the boundaries**: \$34,999 is now treated as utterly different from \$35,000, yet \$35,000 and \$100,000 fall in the same bin. Choosing good cut points is non-trivial — plot histograms, lean on **quantiles**, common sense, and subject-matter expertise. The author admits discretization has **rarely helped** in practice.
:::

### Encoding Categorical Features

Newcomers assume categories are **static** — a fixed, known set. Often true: age brackets and income brackets won't change, so you just number each one. But in production, **categories change.** Imagine an Amazon recommender that uses **product brand** as a feature. Even in 2019 there were over **two million** brands, and new ones appear constantly. Watch the cascade of "fixes" and how each one fails:

1. Number all 2M brands `0…1,999,999`. Great on the historical test set — then **crashes** in production on a brand it has never seen.
2. Add an `UNKNOWN = 2,000,000` bucket. No more crashes — but the model never saw `UNKNOWN` in training, so it **recommends nothing** from new brands, and sellers complain.
3. Encode the top 99% of brands and lump the bottom 1% as `UNKNOWN` so the model *learns* what `UNKNOWN` means. Works for an hour — then click-through collapses, because brand-new **luxury, knockoff, and established** brands all get treated like *unpopular* brands.

This is not an Amazon-only problem. New accounts, product types, domains, restaurants, companies, IP addresses — anything with an open-ended category space hits it.

The elegant fix is the **hashing trick** (popularized by Microsoft's *Vowpal Wabbit*). Run each category through a **hash function** and use the hash as its index. Because you choose the **hash space** up front, you fix the number of possible indices *without knowing how many categories will ever exist*. Pick an 18-bit space → $2^{18} = 262{,}144$ slots, and **every** category — even ones never seen in training — lands on an index in `0…262,143`.

The obvious worry is **collisions** — two categories landing on the same index. Three reassurances:

- Collisions are **random**: a new brand might share a slot with *any* existing brand, not always with the unpopular ones — a strict improvement over the `UNKNOWN`-bucket approach.
- The damage is **tiny**: Booking.com found that even at a **50% collision rate**, log loss rose by **less than 0.5%**.
- You can shrink collisions by enlarging the hash space, or pick a **locality-sensitive** hash so similar categories (e.g., look-alike website names) land *near* each other.

Academics often dismiss the hashing trick as "hacky," but its industry adoption speaks for itself — it ships in scikit-learn, TensorFlow, and gensim, and it's especially handy in **continual learning**, where the model keeps meeting brand-new categories in live traffic.

### Feature Crossing

**Feature crossing** combines two or more features into a new one, to capture **nonlinear relationships**. Suspect that *marital status* and *number of children* interact in a non-additive way? Cross them into a single `marriage_and_children` feature with values like `("Single", 0)`, `("Married", 2)`, `("Single", 1)`.

Crossing matters most for models that **can't model nonlinearity on their own** — linear regression, logistic regression, tree-based models. Neural networks need it less, but **explicit** crosses can still help them learn faster; the **DeepFM / xDeepFM** family exploits exactly this for recommendation and click-through prediction.

::: callout warning "Crossing can blow up your feature space"
Cross a 100-value feature with another 100-value feature and you get **10,000** possible values. That demands far more data to learn well, and the extra dimensionality makes the model more prone to **overfitting**.
:::

### Discrete and Continuous Positional Embeddings

Introduced to deep learning by *"Attention Is All You Need"* (Vaswani et al., 2017), **positional embeddings** are now standard across NLP and computer vision. The motivation: in **language modeling** you predict the next token from a sequence of previous tokens. A **recurrent network** reads tokens in order, so position is *implicit*. A **transformer** reads them in **parallel**, so you must feed position **explicitly** — otherwise *"a dog bites a child"* and *"a child bites a dog"* look identical to the model.

You can't just feed the raw integers `0, 1, …, 7`: neural networks dislike inputs that aren't unit-variance (the same reason we scale features). But rescaling them to `0, 0.143, …, 1` makes adjacent positions **too close** to tell apart. Two better approaches exist:

::: callout key "Embeddings, learned vs. fixed"
An **embedding** is a vector that represents a piece of data; the set of all such vectors from one algorithm is an **embedding space**, and they're all the same size. **Word embeddings** are the classic case, but there are embeddings for products (Criteo, Coveo), and for images, graphs, queries, and users (Pinterest) — sparking interest in **universal** multimodal embeddings.

- **Learned positional embeddings.** Treat position like a word: build an embedding matrix whose columns are positions `0…7`. Position and word embeddings share a size so they can be **summed** — the vector for *"food" at position 0* is `embed("food") + embed(position 0)`. As model weights update, these vectors change, so they're **learned**. (This is how Hugging Face's BERT did it.)
- **Fixed positional embeddings.** Each position's vector is **predefined** by a function — typically **sine** at even indices, **cosine** at odd. Nothing is trained.
:::

Fixed positional embedding is a special case of **Fourier features**. Crucially, while embedding-matrix positions must be **discrete**, Fourier features can be **continuous** — vital for tasks like representing a **3D object** (e.g., a teapot), where every surface point is a continuous 3-D coordinate. You can't index an embedding matrix with continuous columns, but sine/cosine still work. The general **Fourier features** of a coordinate $v$ are:

$$\gamma(v) = \big[\, a_1\cos(2\pi b_1^\top v),\; a_1\sin(2\pi b_1^\top v),\; \dots,\; a_m\cos(2\pi b_m^\top v),\; a_m\sin(2\pi b_m^\top v) \,\big]^\top$$

These have been shown to improve models that take coordinates or positions as input (Tancik et al., 2020).

## Data Leakage

In July 2021, *MIT Technology Review* ran the headline *"Hundreds of AI Tools Have Been Built to Catch Covid. None of Them Helped."* Several models scored beautifully in evaluation yet were useless in practice. One was trained on a mix of scans of patients **lying down** and **standing up** — and since the sickest patients were scanned lying down, the model learned to **predict serious Covid from body position**. Others latched onto the **text font** a hospital used to label scans, so fonts from overloaded hospitals became "predictors" of risk.

::: callout key "What is data leakage?"
**Data leakage** is when a form of the **label sneaks into the features** during training, but **isn't available at inference time**. The model learns to "cheat" off information it won't have in production — so it aces evaluation and then fails spectacularly in the wild. Leakage is **dangerous** because it's usually **non-obvious** and survives extensive testing, and it's rarely taught in ML courses.
:::

A second canonical example: a lung-cancer CT model trained on **hospital A** data (doctors' diagnoses removed) did great on hospital A's test set but **poorly on hospital B**. The cause: at hospital A, patients suspected of cancer were sent to a **fancier scanner** that produced subtly different images, so the model learned to read the **machine**, not the disease. Hospital B assigned scanners at random, leaving nothing to exploit. The label had leaked into the features. (A 2020 Kaggle Ion-Switching competition saw the *opposite* abuse: teams reverse-engineered the test labels from a leak and **won** by exploiting it.)

Leakage creeps in through at least six channels:

- **Splitting time-correlated data randomly instead of by time.** When the *time* data is generated affects its label distribution, a random split scatters **future** samples into training. With stock prices (similar stocks move together), putting day 7 in the train split leaks that day's market mood. The effect can be **non-obvious** too: if an artist dies, listening spikes that day, so any same-day sample in training makes other same-day predictions artificially easy. **Fix:** split **by time** — e.g., train on weeks 1–4, then randomly split week 5 into validation and test.
- **Scaling before splitting.** Computing global mean/variance on the *whole* dataset before splitting leaks the test set's statistics into training. **Fix:** **split first**, compute statistics from the **train split only**, then apply them to every split. Some go further and split before *any* exploratory analysis.
- **Filling missing values with test-set statistics.** Same bug, different operation: imputing with a mean/median computed over all the data leaks test information. **Fix:** impute using **train-split statistics** only.
- **Poor deduplication before splitting.** Duplicates or near-duplicates that survive into both train and test inflate your score. It's common even in famous datasets — Barz & Denzler (2019) found **3.3%** of CIFAR-10 and **10%** of CIFAR-100 test images had duplicates in the training set. Merging datasets (one Covid dataset unknowingly *contained* another) and oversampling both create duplicates. **Fix:** dedupe **before** splitting (and check again after), and **oversample only after** splitting.
- **Group leakage.** Strongly correlated examples land in different splits. A patient's two CT scans a week apart share a label, yet one falls in train and the other in test; the same happens with burst photos of one object taken milliseconds apart. Hard to avoid without understanding how the data was generated.
- **Leakage from the data-collection process.** The lung-cancer-via-scanner story is this type: the *procedure* that produced the data carried the label. **Mitigations:** track data **sources**, **normalize** so different sources share means/variances (e.g., resize all scans to one resolution so the model can't tell machines apart), and bring in **subject-matter experts** who understand collection.

### Detecting Data Leakage

Leakage can enter at *any* step — generating, collecting, sampling, splitting, processing, or feature-engineering — so monitor for it across the whole project lifecycle.

- **Measure each feature's predictive power** against the label. An unusually high correlation is a red flag: investigate how the feature is generated and whether the correlation *makes sense*. Watch for **pairwise** leakage too — a start date and an end date are each innocent, but **together** they reveal tenure exactly.
- **Run ablation studies.** Remove a feature (or a set) and see how much performance drops; a feature whose removal craters the model deserves scrutiny. With thousands of features you can't ablate every combination, but you can probe the **suspicious** ones. Ablations run **offline** on your own schedule — good use of idle machines.
- **Be suspicious of any new feature that suddenly boosts performance.** Either it's genuinely great, or it's smuggling in leaked label information.
- **Treat the test split as radioactive.** Use it for *anything* other than reporting final performance — generating feature ideas, tuning hyperparameters — and you risk leaking the future into training.

## Engineering Good Features

Generally, more features help — in practice the feature list for a production model only **grows**. But **more is not always better**, and too many features hurt during both training and serving:

::: callout warning "The cost of too many features"
- More features → **more chances for data leakage.**
- More features → **overfitting.**
- More features → **more memory** to serve (pricier instances).
- More features → **higher inference latency**, especially when features must be extracted from raw data online.
- Useless features become **technical debt**: every pipeline change forces you to update all features that depend on it (drop "age" from the app and every age-derived feature must change).
:::

In theory, **L1 regularization** should drive a useless feature's weight to 0. In practice, *removing* dead or harmful features explicitly helps the model learn faster. You can stash removed features (or just their **definitions**) to revive or share later — and yes, **feature stores** can help manage definitions, but that's Chapter 10. Two lenses decide whether a feature earns its place: **importance** and **generalization**.

### Feature Importance

There are many ways to measure importance. For classical models like **boosted gradient trees**, use the **built-in importance** functions (e.g., XGBoost's `get_score`). For a **model-agnostic** view, reach for **SHAP** (SHapley Additive exPlanations); **InterpretML** is another strong open-source option.

The exact math is involved, but the intuition is simple: **a feature's importance is how much performance drops when you remove it.** SHAP's superpower is granularity — it scores a feature's importance to the **whole model** *and* its contribution to a **single prediction**.

A recurring pattern: a handful of features dominate. Measuring importance for a click-through model, Facebook's ads team found the **top 10 features drove ~half** of total importance, while the **bottom ~300 contributed under 1%**. Beyond feature selection, importance scores double as a powerful **interpretability** tool.

### Feature Generalization

Since the whole point is correct predictions on **unseen** data, features must **generalize**. Not all do equally: a comment's **ID** is useless (it never recurs), but the **user's** identifier might genuinely help. Measuring generalization is less scientific than measuring importance — it leans on intuition and domain expertise alongside statistics — and it has two aspects.

**Coverage** — the share of samples that *have* a value for the feature (fewer blanks → higher coverage). Rough rule: a feature present in only a sliver of your data probably won't generalize (e.g., *number of children* available for just 1% of rows). But it's only a *rough* rule — a sparse feature can still be gold if its missingness is **MNAR**: if a feature appears in 1% of data yet **99% of those rows are POSITIVE**, use it. Also watch coverage **across slices and over time**: if a feature covers 90% of the train split but only 20% of test, your splits likely **aren't from the same distribution** — investigate the split and whether this feature is leaking.

**Distribution of values** — even when present, the *values* must line up across splits. If the values in the seen data **don't overlap** with those in unseen data, the feature can actively **hurt**.

::: callout example "Taxi-ride ETA: DAY_OF_THE_WEEK vs HOUR_OF_THE_DAY"
Retraining weekly on the last six days to predict today's ETAs, `DAY_OF_THE_WEEK` has **100% coverage** — but the train split holds Monday–Saturday while the test day is **Sunday**, a value the model never saw. Without a clever encoding, it won't generalize and may hurt performance. `HOUR_OF_THE_DAY` is a *great* feature: hour affects traffic and its train/test value ranges overlap **100%**.

There's a **generalization-vs-specificity** trade-off, too. You could replace hour with `IS_RUSH_HOUR` (1 between 7–9 a.m. or 4–6 p.m.). It's *more generalizable but less specific* — and using it **instead of** `HOUR_OF_THE_DAY` discards real information about the hour.
:::

## Summary

Because today's ML systems still rise or fall on their features, any organization serious about production ML should invest real time in feature engineering. There's no foolproof recipe — the best teacher is **experience** (try features, watch what they do), supplemented by learning from experts (Kaggle write-ups are a goldmine). Feature work often needs **subject-matter expertise** from people who aren't engineers, so design a workflow that lets non-engineers contribute. And the work never truly ends: in real projects, data collection and feature engineering continue for as long as the model is in production, feeding **continual learning** (Chapter 9).

## Key takeaways

::: takeaways
- **Features beat algorithms.** Once a model works, better features usually give the biggest gain — and even SOTA architectures fail on bad features. Deep learning automates feature extraction for *text and images*, but most production ML still needs hand-engineered features.
- **Missing values aren't all alike.** Classify them as **MCAR / MAR / MNAR**, because the type dictates whether deletion biases your model and whether the blank is itself a signal. There is **no perfect** fix; deletion loses info, imputation can inject bias or leakage. Never impute with a value that's also a legitimate reading (don't fill age with 0).
- **Scale features**, computing statistics on the **train split only**; use min-max, standardization, or a log transform for skew. **Bin** continuous values when data is scarce, accepting hard boundary edges.
- **Encode open-ended categories with the hashing trick** — a fixed hash space absorbs categories unseen at training time. Collisions are random and cheap (50% collisions ≈ <0.5% loss).
- **Cross features** to model nonlinearity (great for linear/tree models), and use **positional embeddings / Fourier features** to give transformers a sense of order — discrete via an embedding matrix, continuous via sine/cosine.
- **Data leakage** = label information sneaking into features but absent at inference. Top causes: random splits of **time-correlated** data, **scaling/imputing before splitting**, poor **dedup**, **group leakage**, and the **collection process** itself. Detect it via per-feature predictive power, **ablation studies**, and healthy suspicion of any feature that suddenly boosts performance. **Never** touch the test split except to report final numbers.
- **Judge features by importance (SHAP, tree importances) and generalization (coverage + value-range overlap across splits).** More features means more leakage risk, overfitting, latency, and tech debt — so remove the dead ones.
:::

Q: Distinguish **MCAR, MAR, and MNAR**.
